## アジェンダ

### 1. イントロダクション（15分）

- ワークショップ「アーキテクチャ登竜門」の趣旨説明
- 「アーキテクチャ」とは何か — ソフトウェアにおける設計判断の集合
- 本ワークショップで扱う範囲と扱わない範囲

> **講師メモ**: 「アーキテクチャ」という言葉は抽象的に聞こえがちです。まずは「ソフトウェアの構造に関する、後から変えにくい判断のこと」という定義を共有しましょう。Ralph Johnson の「アーキテクチャとは重要なもののこと。それが何であれ」という言葉を引用すると、参加者が「設計判断の重み」をイメージしやすくなります。
>
> 扱う範囲（アプリケーションアーキテクチャ、データ設計、分散システム）と扱わない範囲（インフラ構築の実務、フロントエンドフレームワーク固有の設計、セキュリティの深掘り）を最初に明確にしておくと、参加者の期待値が揃います。

### 3. 全体ロードマップの紹介（20分）

- 9つのテーマの概要と順序の意図

#### 3つのフェーズの意図

本ワークショップは3つのフェーズに分けて構成されています。

```
フェーズ1: 基礎（第1〜3回）
  オブジェクト指向 → RDB → データモデリング
  「1つのアプリケーション内のコードとデータの設計」を固める

フェーズ2: 設計パターン（第4〜6回）
  デザインパターン → DDD → クリーンアーキテクチャ
  「設計の引き出しを増やし、アプリケーション全体の構造を設計する力」を養う

フェーズ3: 分散システム（第7〜9回）
  マイクロサービス → オーケストレーション → イベント駆動
  「複数のアプリケーション（サービス）にまたがるシステム全体の設計」を学ぶ
```

フェーズが進むにつれて**スコープが広がる**構成です。最初は関数やクラスの設計から始まり、最終的にはシステム全体のアーキテクチャ判断ができるようになることを目指します。順序に意味があるため、飛ばさず順番に受講することを推奨します。

### 6. ウォームアップ：設計について考える（15分）

以下の問いについて、グループで議論する。

- 「良いコード」と「良い設計」の違いは何か？
- 設計で一番困った経験は？
- なぜアーキテクチャを学ぶ必要があるのか？

> **ファシリテーションのコツ**: この議論に正解はありません。重要なのは参加者が「設計について言語化する」体験をすることです。以下のような観点を引き出せると理想的です。
>
> - **良いコード** = 読みやすい、バグが少ない、テストしやすい（局所的な品質）
> - **良い設計** = 変更しやすい、拡張しやすい、チームで開発しやすい（構造的な品質）
> - 「コードは正しく動くが、設計が悪いと新機能追加のたびに苦しむ」という実体験を引き出しましょう
>
> 出てきた意見はホワイトボードに書き留め、最終回（第10回）で振り返ると、参加者の成長が可視化できます。
