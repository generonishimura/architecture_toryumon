## 講義内容

### 1. DDDの概要と背景（15分）

- Eric Evans『Domain-Driven Design』（2003）の本質
- DDDは「技術的な手法」ではなく「ビジネスとの向き合い方」
- DDDが解決する問題 — ビジネスとコードの乖離
- DDD = 戦略的設計 + 戦術的設計

> **よくある誤解**: 「DDDとはエンティティや値オブジェクトを使うこと」と思われがちですが、それは戦術的設計（実装パターン）に過ぎません。DDDの本質は**戦略的設計** — つまり「ビジネスの複雑さをどう理解し、どうモデル化するか」にあります。
>
> DDDが特に効果を発揮するのは、ビジネスロジックが複雑なドメイン（金融、物流、医療など）です。CRUD中心のシンプルなアプリケーションにDDDを適用するのは過剰です。

```
DDDの全体像:

  戦略的設計（チーム・組織レベル）
  ├── ユビキタス言語: ビジネスとコードで同じ言葉を使う
  ├── 境界づけられたコンテキスト: 用語の意味が一貫する範囲を区切る
  └── コンテキストマップ: コンテキスト間の関係を明確にする

  戦術的設計（コードレベル）
  ├── エンティティ: 識別子を持つオブジェクト
  ├── 値オブジェクト: 値そのものが意味を持つオブジェクト
  ├── 集約: 整合性の境界
  ├── ドメインサービス: エンティティに属さないロジック
  ├── ドメインイベント: 起こった事実の記録
  └── リポジトリ: 永続化の抽象
```

### 2. 戦略的設計（30分）

#### ユビキタス言語（Ubiquitous Language）

- チーム全員（開発者 + ドメインエキスパート）が共通で使う言葉
- コード上の名前もユビキタス言語に従う
- 例：「注文」「配送」「請求」— ビジネスの言葉がそのままコードに

```typescript
// Bad: 技術者都合の命名 — ビジネスサイドに通じない
class DataProcessor {
  processItem(record: Record): Result { /* ... */ }
  updateFlag(id: string, flag: boolean): void { /* ... */ }
}

// Good: ユビキタス言語に基づく命名 — ビジネスサイドと同じ言葉
class OrderService {
  placeOrder(cart: ShoppingCart): Order { /* ... */ }
  cancelOrder(orderId: OrderId, reason: CancellationReason): void { /* ... */ }
}
```

> **ユビキタス言語を作る実践的なステップ**:
> 1. ビジネスサイドとの会話で使われる用語を収集する
> 2. 用語集（Glossary）を作成し、定義を明文化する
> 3. コード上のクラス名・メソッド名を用語集と一致させる
> 4. 新しい用語が出てきたら用語集を更新し、コードにも反映する

#### 境界づけられたコンテキスト（Bounded Context）

- 同じ用語が異なる意味を持つ場合の境界の設定
- 例：「商品」は在庫管理と商品カタログで意味が異なる
- コンテキストマップ — コンテキスト間の関係の可視化
- 関係パターン：共有カーネル、顧客/供給者、腐敗防止層

> **「商品」が文脈によって意味が変わる例**:
>
> | コンテキスト | 「商品」の意味 | 持つ属性 |
> |------------|-------------|---------|
> | 商品カタログ | 顧客に見せるもの | 名前、説明、画像、カテゴリ、価格 |
> | 在庫管理 | 倉庫にある物理的なもの | SKU、在庫数、倉庫場所、ロット番号 |
> | 注文管理 | 注文された品目 | 商品ID、注文数量、注文時点の価格 |
> | 配送管理 | 配送する荷物 | 重量、サイズ、取扱注意フラグ |
>
> これらを1つの `Product` クラスで表現しようとすると、巨大で複雑なクラスになります。コンテキストごとに分離することで、各コンテキスト内ではシンプルなモデルを維持できます。

#### コンテキスト間の関係パターン

| パターン | 説明 | 使う場面 |
|---------|------|---------|
| **共有カーネル** | 2つのコンテキストがモデルの一部を共有 | 密に連携するチーム間で使用。変更は両者の合意が必要 |
| **顧客/供給者** | 上流（供給者）が下流（顧客）のニーズを聞いて開発 | 別チームだが協力的な関係 |
| **順応者** | 下流が上流のモデルにそのまま従う | 上流が変更に応じてくれない場合 |
| **腐敗防止層 (ACL)** | 下流が変換層を設けて上流の影響を遮断 | レガシーシステムや外部APIとの接続 |
| **公開ホストサービス** | 上流がAPI等の標準的なインターフェースを公開 | 複数の下流に対応する場合 |

#### コンテキストマップの実践

```
┌──────────────────┐     ┌──────────────────┐
│  商品カタログ      │     │  在庫管理         │
│  ──────────       │     │  ──────          │
│  商品: 名前、説明、│ ACL │  商品: SKU、数量、│
│  カテゴリ、価格    │◄───│  倉庫、入出庫     │
│                   │     │                  │
└──────────────────┘     └──────────────────┘
         │                         │
         │ 公開ホスト               │ 順応者
         ▼                         ▼
┌──────────────────┐     ┌──────────────────┐
│  注文管理         │     │  配送管理         │
│  ──────          │     │  ──────          │
│  注文: 明細、合計、│────▶│  配送: 宛先、     │
│  ステータス       │     │  追跡番号、状態   │
└──────────────────┘     └──────────────────┘
```

### 3. 戦術的設計（40分）

#### エンティティ（Entity）

- 一意の識別子を持ち、ライフサイクルを通じて同一性を保つ
- 属性が変わっても「同じもの」であると判断できる
- TypeScript例：`Order` エンティティ

> **エンティティの見分け方**: 「名前が変わっても同じものか？」と問いかけます。ユーザーが名前を変更しても「同じユーザー」→ エンティティ。一方、住所が「東京都港区→大阪市北区」に変わったら「別の住所」→ 値オブジェクト。

#### 値オブジェクト（Value Object）

- 識別子を持たず、属性の値そのものが意味を持つ
- イミュータブル（不変）であること
- TypeScript例：`Money`、`EmailAddress`、`Address`

```typescript
// 値オブジェクトの追加例: EmailAddress
class EmailAddress {
  private constructor(readonly value: string) {}

  static of(email: string): EmailAddress {
    if (!email.includes('@') || email.length > 255) {
      throw new Error(`無効なメールアドレスです: ${email}`);
    }
    return new EmailAddress(email.toLowerCase()); // 正規化
  }

  get domain(): string {
    return this.value.split('@')[1];
  }

  equals(other: EmailAddress): boolean {
    return this.value === other.value;
  }

  toString(): string {
    return this.value;
  }
}

// プリミティブ型(string)ではなく値オブジェクトを使う利点:
// 1. バリデーションが一箇所に集約される
// 2. 不正な値が存在できない（コンストラクタで検証）
// 3. ドメインに関連する操作（domain取得等）を値オブジェクト自身が持てる
// 4. 型レベルで「メールアドレス」と「ただの文字列」を区別できる
```

> **値オブジェクトを使うべき判断基準**: プリミティブ型（`string`、`number`）に「バリデーションルール」や「関連する操作」がある場合は、値オブジェクトに昇格させましょう。金額、メール、電話番号、日付範囲などが典型です。

#### 集約（Aggregate）

- 関連するエンティティと値オブジェクトのまとまり
- 集約ルート（Aggregate Root）を通じてのみ操作する
- トランザクション整合性の境界
- 集約の設計指針 — 小さく保つ

```
【注文の集約の例】

┌─────────────────────────────────┐
│  Order（集約ルート）              │
│  ├── OrderItem（エンティティ）     │  ← Orderを通じてのみアクセス
│  │   └── Money（値オブジェクト）   │
│  ├── Address（値オブジェクト）     │
│  └── OrderStatus（値オブジェクト） │
└─────────────────────────────────┘

ルール:
・外部から OrderItem を直接操作しない
・Order.addItem() や Order.confirm() を通じて操作する
・1つの集約 = 1つのトランザクション境界
```

> **集約を小さく保つ理由**: 集約が大きいと、1つの操作で大量のデータをロック・更新する必要が生じ、パフォーマンスと並行性の問題が起きます。Vaughn Vernon は「集約は1つのルートエンティティと、最小限の関連オブジェクトで構成すべき」と推奨しています。
>
> **よくある失敗**: `Customer` 集約に `Order[]` を含めてしまう → 顧客の注文が増えるたびに集約全体をロードする必要が生じる。正しくは `Order` を独立した集約にし、`customerId` で参照する。

#### ドメインサービス

- エンティティや値オブジェクトに属さないドメインロジック
- ステートレスな操作
- TypeScript例：送金処理、在庫引当

#### ドメインイベント

- ドメイン内で発生した重要な出来事を表現
- 「〇〇が起こった」という過去形で命名
- TypeScript例：`OrderPlaced`、`PaymentCompleted`

#### リポジトリ（再掲・深掘り）

- 第4回で学んだRepositoryパターンのDDD文脈での役割
- 集約ルート単位で永続化と取得を行う

## コード例の概要

```typescript
// 値オブジェクト
class Money {
  private constructor(
    readonly amount: number,
    readonly currency: string
  ) {
    if (amount < 0) throw new Error('金額は0以上である必要があります');
  }

  static of(amount: number, currency: string = 'JPY'): Money {
    return new Money(amount, currency);
  }

  add(other: Money): Money {
    if (this.currency !== other.currency) {
      throw new Error('通貨が異なります');
    }
    return Money.of(this.amount + other.amount, this.currency);
  }

  equals(other: Money): boolean {
    return this.amount === other.amount && this.currency === other.currency;
  }
}
```

```typescript
// エンティティ（集約ルート）
class Order {
  private _items: OrderItem[] = [];
  private _status: OrderStatus = OrderStatus.DRAFT;

  constructor(
    readonly id: string,
    readonly customerId: string,
    private _shippingAddress: Address
  ) {}

  addItem(productId: string, quantity: number, unitPrice: Money): void {
    if (this._status !== OrderStatus.DRAFT) {
      throw new Error('確定済みの注文には商品を追加できません');
    }
    this._items.push(new OrderItem(productId, quantity, unitPrice));
  }

  get totalAmount(): Money {
    return this._items.reduce(
      (sum, item) => sum.add(item.subtotal),
      Money.of(0)
    );
  }

  confirm(): OrderPlaced {
    if (this._items.length === 0) {
      throw new Error('商品が1つもない注文は確定できません');
    }
    this._status = OrderStatus.CONFIRMED;
    // ドメインイベントを返す
    return new OrderPlaced(this.id, this.customerId, this.totalAmount);
  }

  get items(): ReadonlyArray<OrderItem> {
    return [...this._items];
  }

  get status(): OrderStatus {
    return this._status;
  }
}
```

```typescript
// ドメインイベント
class OrderPlaced {
  readonly occurredAt: Date = new Date();

  constructor(
    readonly orderId: string,
    readonly customerId: string,
    readonly totalAmount: Money
  ) {}
}
```

### コード課題

「ホテル予約システム」のドメインモデルをTypeScriptで設計してください。

要件：
- 宿泊客が客室を予約できる
- 客室には種類（シングル、ダブル、スイート）と料金がある
- 予約にはチェックイン日・チェックアウト日がある
- 同じ客室の同一日程に複数の予約はできない
- 予約をキャンセルできる（チェックイン前日まで）

成果物：
- 値オブジェクト、エンティティ、集約の識別
- TypeScriptのコード（ドメインモデルの実装）
- ビジネスルールがドメインモデル内に表現されていること
