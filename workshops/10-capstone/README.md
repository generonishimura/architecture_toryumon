# 第10回：総合演習・振り返り

> 学んだ知識を統合し、実践的な設計判断力を確認する

## 基本情報

| 項目 | 内容 |
|------|------|
| 所要時間 | 2.5〜3時間 |
| 前提知識 | 第1回〜第9回のすべての内容 |
| 到達目標 | 全テーマの知識を統合し、アーキテクチャの設計判断を根拠を持って説明できる |

## この回のキーメッセージ

> 「正解は1つではない。トレードオフを理解し、根拠を持って判断することが大切」

## アジェンダ

### 1. 全体振り返り（30分）

#### 9つのテーマの総復習

各テーマの核心メッセージを振り返る。

| フェーズ | テーマ | 核心 |
|---------|--------|------|
| 基礎 | オブジェクト指向 | 責務の明確化と依存の制御 |
| 基礎 | RDB | データ整合性の保証 |
| 基礎 | データモデリング | ビジネス構造のモデル化 |
| 設計パターン | デザインパターン | 問題に応じた設計の型の選択 |
| 設計パターン | DDD | ビジネスの言葉でコードを書く |
| 設計パターン | クリーンアーキテクチャ | ビジネスロジックを外部から守る |
| 分散システム | マイクロサービス | 必要に応じた適切な分割 |
| 分散システム | オーケストレーション | 自動化と標準化による運用 |
| 分散システム | イベント駆動 | 非同期通信と結果整合性 |

#### テーマ間のつながりの再確認

```
オブジェクト指向（SOLID）
    ├── デザインパターン（設計の型）
    │       └── DDD（ビジネスの構造化）
    │               └── クリーンアーキテクチャ（アプリ構造）
    │                       └── マイクロサービス（サービス分割）
    │                               ├── オーケストレーション（運用）
    │                               └── イベント駆動（通信）
    └── RDB + データモデリング（データの構造化）
            └── DDD（永続化、リポジトリ）
                    └── マイクロサービス（Database per Service）
```

### 2. 総合演習：アーキテクチャ設計レビュー（90分）

#### 演習の概要

架空のシステムの要件に対して、グループでアーキテクチャを設計する。

#### 題材例：オンライン学習プラットフォーム

> ユーザーがコースを受講し、動画を視聴し、テストを受けて修了証を取得できるプラットフォーム

要件：
- ユーザー登録・認証
- コースカタログの検索・閲覧
- 動画のストリーミング配信
- 進捗管理（どこまで視聴したか）
- テストの受験と自動採点
- 修了証の発行
- 月額サブスクリプション課金
- 管理者によるコース・ユーザー管理

#### 演習の進め方（90分）

```
[15分] グループ分け（3〜4人）、要件の読み合わせ

[30分] 設計ワーク
       以下の観点で設計を行う：
       - ドメインモデルの設計（エンティティ、値オブジェクト、集約）
       - 境界づけられたコンテキストの識別
       - サービス分割の判断（モノリス / モジュラーモノリス / マイクロサービス）
       - データモデルの設計
       - サービス間通信の方式
       - 同期 vs 非同期の判断

[30分] 各グループの発表（1グループ10分）

[15分] 講師からのフィードバック・全体ディスカッション
```

#### 設計ワークのガイド: 段階的に考える

設計ワークでは、以下のステップで段階的に考えるとスムーズです。

```
Step 1: ドメインを理解する（5分）
  ・登場するエンティティを洗い出す
  ・ユビキタス言語を定義する
  ・「ユーザー」「コース」「テスト」「修了証」等

Step 2: 境界を引く（5分）
  ・同じ言葉でも文脈が異なるものを探す
  ・例: 「ユーザー」は認証コンテキストでは「アカウント」、
        学習コンテキストでは「受講者」
  ・境界づけられたコンテキストを識別

Step 3: アーキテクチャスタイルを決める（5分）
  ・チーム規模、スケーリング要件、開発速度を考慮
  ・最初はモジュラーモノリスで十分か？
  ・動画配信だけ独立させるべきか？

Step 4: データモデルを設計する（10分）
  ・主要なエンティティの属性と関連を定義
  ・Database per Service にする場合、データの境界はどこか

Step 5: 通信方式を決める（5分）
  ・同期: コースカタログの検索、テストの採点結果
  ・非同期: 修了証の発行、メール通知
```

#### 評価ルーブリック（講師用）

| 観点 | 優秀 | 良好 | 要改善 |
|------|------|------|--------|
| ドメイン理解 | エンティティと値オブジェクトが正しく識別され、ビジネスルールが表現されている | 主要なエンティティが識別されている | エンティティの識別が不十分 |
| 境界設計 | コンテキスト境界が明確で、境界間の関係パターンが説明されている | コンテキスト境界が識別されている | 境界が不明確、または1つの巨大なコンテキスト |
| アーキテクチャ判断 | 選択理由が具体的で、トレードオフが明示されている | 選択理由が説明されている | 理由なく「マイクロサービスにした」等 |
| データ設計 | 正規化、インデックス、データ分離が適切 | 主要テーブルが設計されている | テーブル設計が不十分 |
| 通信設計 | 同期/非同期の使い分けが根拠を持って説明されている | 通信方式が決められている | 通信方式が未検討 |

#### 発表時のポイント

- **「何を選んだか」** ではなく **「なぜそれを選んだか」** を重視
- トレードオフを明示する（「〇〇を得るために△△を犠牲にした」）
- 第1回〜第9回で学んだ概念を使って説明する

##### 実践の場

- **社内勉強会の継続** — 定期的にアーキテクチャの議論を行う
- **設計レビューの習慣化** — PRレビューで設計観点のフィードバックを行う
- **ADR（Architecture Decision Record）** — 設計判断を記録する文化の導入
- **自チームのコードへの適用** — 学んだ概念を段階的に導入

> **段階的な導入のススメ**: 学んだ概念をすべて一度に適用しようとすると、チームに受け入れられません。以下の順序で段階的に導入することを推奨します。
>
> ```
> Phase 1（すぐに始められる）:
>   ・PRレビューで SOLID原則の観点を1つ追加する
>   ・新機能の設計時にADRを1つ書いてみる
>   ・テストが書きにくいコードを見つけたら、依存性逆転を適用する
>
> Phase 2（チームで合意して導入）:
>   ・リポジトリパターンの導入（データアクセスの抽象化）
>   ・ユースケースクラスの導入（ビジネスロジックの分離）
>   ・ユビキタス言語の用語集を作成し、コード上の命名を統一する
>
> Phase 3（プロジェクト規模に応じて検討）:
>   ・クリーンアーキテクチャのレイヤー構造の導入
>   ・ドメインモデルの本格的な設計
>   ・マイクロサービスへの段階的な移行
> ```

##### 本ワークショップで扱わなかったテーマ

今後の学習として以下のテーマもお勧めします。

- **セキュリティアーキテクチャ** — 認証・認可（OAuth 2.0、OIDC）
- **パフォーマンスエンジニアリング** — キャッシュ戦略、CDN
- **モニタリング・SRE** — SLI/SLO、インシデント対応
- **データエンジニアリング** — データパイプライン、データレイク
- **AI/MLシステムの設計** — モデルサービング、特徴量ストア

## 最終課題（任意・ワークショップ完了後）

### 個人プロジェクト

自分の所属チームが開発しているシステム（の一部）について、以下を行ってください。

1. **現状分析**
   - 現在のアーキテクチャをドキュメント化する
   - ドメインモデルを描く
   - 改善すべき点を洗い出す

2. **改善提案**
   - 改善案を設計する
   - 改善のトレードオフを明示する
   - 段階的な移行計画を立てる

3. **ADRの作成**
   - 主要な設計判断をADR形式で記録する
   - テンプレート：

```markdown
# ADR-001: [タイトル]

## ステータス
提案中 / 承認済み / 廃止

## コンテキスト
どのような状況でこの判断が必要になったか

## 決定
何を決定したか

## 理由
なぜその決定をしたか（代替案との比較を含む）

## 結果
この決定によって生じる影響（良い面・悪い面）
```
