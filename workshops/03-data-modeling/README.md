# 第3回：データモデリング

> ビジネスの構造を正しく反映したデータモデルを設計する

## 基本情報

| 項目 | 内容 |
|------|------|
| 所要時間 | 2〜2.5時間 |

## この回のキーメッセージ

> 「良いデータモデルは、ビジネスの構造を正しく反映している」

## 講義内容

### 1. データモデリングとは（10分）

- 現実世界の構造をデータとして表現する技法
- なぜ「いきなりテーブル設計」ではダメなのか
- データモデリングの3段階：概念 → 論理 → 物理

> **なぜ「いきなりCREATE TABLE」ではダメなのか？**
> テーブル設計から始めると、技術的な制約（DBの型、インデックス）に引きずられて、ビジネスの本質的な構造を見落としがちです。例えば「ユーザーと注文の関係は？」「注文のステータス遷移はどうなる？」といったビジネス上の問いに先に答えることで、正しいテーブル設計にたどり着けます。
>
> ```
> ❌ よくある失敗パターン:
>    「とりあえずテーブル作るか」→ 後からカラム追加の嵐 → 設計がカオスに
>
> ✅ 正しいアプローチ:
>    概念モデル（何が登場する？）→ 論理モデル（どう関連する？）→ 物理モデル（どう格納する？）
> ```

### 2. 概念データモデル（20分）

#### 概念モデルの目的

- ビジネスの主要なエンティティとその関係を洗い出す
- 技術的な制約を含まない、ビジネス視点のモデル
- ステークホルダーとの認識合わせに使う

> 概念モデルの読者は「エンジニア」だけではありません。プロダクトマネージャーやビジネスサイドのメンバーとも共有できるレベルのシンプルさが重要です。技術用語（外部キー、インデックス等）はこの段階では一切不要です。

#### ER図の基本要素

- エンティティ（実体）
- アトリビュート（属性）
- リレーションシップ（関連）
- カーディナリティ（多重度）— 1:1、1:N、M:N

```
【カーディナリティの読み方】

  1:1  部門 ──── 部門長      「1つの部門には1人の部門長がいる」
  1:N  部門 ───< 社員        「1つの部門に複数の社員が所属する」
  M:N  学生 >──< 講座        「学生は複数の講座を受講し、講座には複数の学生がいる」

  記号の意味:
  ── : 1側
  <  : 多側（N側）
  >──< : 多対多
```

#### 概念モデリングのコツ

- ビジネスの言葉でエンティティを命名する
- 「〜は〜を持つ」「〜は〜に所属する」で関係を表現
- 最初から完璧を目指さず、繰り返し洗練する

**エンティティの見つけ方**: ビジネス要件の文章から「名詞」を拾い出し、以下で分類します。

| 分類 | 特徴 | 例 |
|------|------|------|
| エンティティ | 独立して存在し、複数のインスタンスがある | 顧客、商品、注文 |
| アトリビュート | エンティティの性質・特徴 | 名前、価格、日付 |
| 関係 | エンティティ間のつながり | 「顧客が注文する」 |
| 範囲外 | システムで管理しないもの | 天候、市場動向 |

### 3. 論理データモデル（20分）

#### 概念モデルから論理モデルへ

- M:N関係の中間テーブルへの分解
- 属性の型の決定（まだ物理的な型ではなく論理的な型）
- 主キーの選定（ナチュラルキー vs サロゲートキー）

```
【M:N関係の分解例】

概念モデル:   学生 >──< 講座     （多対多）
                ↓ 中間テーブルに分解
論理モデル:   学生 ───< 受講 >─── 講座

  学生（学生ID, 名前, メール）
  受講（学生ID, 講座ID, 受講日, 成績）  ← 中間テーブル
  講座（講座ID, 講座名, 単位数）

中間テーブルには「関係そのものの属性」（受講日、成績）を持たせられるのがポイント
```

#### ナチュラルキー vs サロゲートキーの判断

| 観点 | ナチュラルキー（例: ISBN, メールアドレス） | サロゲートキー（例: UUID, auto increment） |
|------|------|------|
| ビジネス上の意味 | あり（業務で使える） | なし（システム内部用） |
| 不変性 | 変わる可能性がある（メール変更等） | 変わらない |
| 一意性の保証 | ビジネスルールに依存 | システムが保証 |
| JOIN性能 | 文字列だと遅くなりがち | 整数なら高速 |
| 推奨場面 | グローバルに一意な標準コード | ほとんどのケースで推奨 |

> **実務での推奨**: 主キーにはサロゲートキー（UUID or auto increment）を使い、ビジネス上のユニークキー（メール、ISBN等）は UNIQUE 制約で別途保証するのが安全です。

#### 論理モデルの設計ポイント

- 正規化の適用（第2回の知識を活用）
- NULL許容の判断基準
- 導出属性（計算で求められる値）をどう扱うか

**NULL許容の判断基準**:
- 「この情報が未設定の状態はビジネス上ありえるか？」で判断
- 例: ユーザーの「電話番号」→ 任意入力なら `NULL` 許容
- 例: 注文の「合計金額」→ 注文が存在する以上、必ずあるべき → `NOT NULL`
- NULLが多いカラムは、設計の見直しサイン（別テーブルに分離すべきかも）

**導出属性の判断**:
- 注文の「合計金額」= 明細の「小計」の合計。保存する？しない？
- 保存しない: 常に最新値が計算される（整合性が高い）
- 保存する: 計算コストを省ける（パフォーマンスが高い）
- 判断基準は「計算コスト」と「更新頻度」のバランス

### 4. 物理データモデル（20分）

#### 論理モデルから物理モデルへ

- データベース固有の型への変換
- インデックス設計（第2回の知識を活用）
- パーティショニングの検討
- 制約の定義（NOT NULL、UNIQUE、CHECK、外部キー）

#### 物理設計の実践的なポイント

- タイムスタンプ列の設計（created_at, updated_at）
- ソフトデリートの是非
- 履歴テーブルの設計パターン
- マイグレーション戦略

**ソフトデリートのメリット・デメリット**:

```typescript
// ソフトデリート: 物理削除せず deleted_at を記録
model User {
  id        String    @id @default(uuid())
  name      String
  email     String    @unique
  deletedAt DateTime? @map("deleted_at")  // NULLなら有効、値ありなら削除済み
  // ...
}
```

| メリット | デメリット |
|---------|----------|
| データ復旧が容易 | 全クエリに `WHERE deleted_at IS NULL` が必要 |
| 監査証跡として活用可能 | UNIQUE制約が複雑になる（削除済みとの重複） |
| 関連データの整合性が壊れない | データ量が増え続ける（定期的なパージが必要） |

> **代替案**: ソフトデリートの代わりに「履歴テーブル」を用意し、削除時に履歴テーブルにコピーしてから物理削除する方法もあります。メインテーブルがクリーンに保たれるため、クエリのパフォーマンスと可読性が向上します。

**履歴テーブルの設計パターン**:

```
パターン1: 別テーブルに履歴を保存（変更ログ方式）
  users テーブル     → 現在の状態のみ
  user_histories    → 変更のたびに旧データをINSERT

パターン2: 同じテーブルに有効期間を持たせる（バイテンポラル方式）
  users テーブルに valid_from / valid_to カラムを追加
  最新レコード: valid_to IS NULL

パターン3: イベントソーシング（第9回で詳しく扱う）
  すべての変更をイベントとして記録し、現在の状態を再構築
```

### 5. 実践：モデリングワークスルー（30分）

講義中に簡単なビジネスドメイン（例：図書管理システム）を題材に、3段階のモデリングを実演する。

- 要件ヒアリングのシミュレーション
- 概念モデルの作成
- 論理モデルへの変換
- 物理モデル（Prisma schema）の作成

## コード例の概要

```typescript
// 概念モデルをPrisma schemaに落とし込む例

// 図書管理システム
model Author {
  id        String   @id @default(uuid())
  name      String
  books     Book[]
  createdAt DateTime @default(now()) @map("created_at")

  @@map("authors")
}

model Book {
  id          String       @id @default(uuid())
  title       String
  isbn        String       @unique
  publishedAt DateTime     @map("published_at")
  authorId    String       @map("author_id")
  author      Author       @relation(fields: [authorId], references: [id])
  loans       BookLoan[]
  createdAt   DateTime     @default(now()) @map("created_at")

  @@index([authorId])
  @@index([isbn])
  @@map("books")
}

model Member {
  id        String     @id @default(uuid())
  name      String
  email     String     @unique
  loans     BookLoan[]
  createdAt DateTime   @default(now()) @map("created_at")

  @@map("members")
}

// M:N を中間テーブルで表現（貸出管理）
model BookLoan {
  id         String    @id @default(uuid())
  bookId     String    @map("book_id")
  memberId   String    @map("member_id")
  loanedAt   DateTime  @default(now()) @map("loaned_at")
  returnedAt DateTime? @map("returned_at")
  book       Book      @relation(fields: [bookId], references: [id])
  member     Member    @relation(fields: [memberId], references: [id])

  @@index([bookId])
  @@index([memberId])
  @@index([returnedAt])
  @@map("book_loans")
}
```
