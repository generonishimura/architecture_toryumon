## 2. Observerパターンの進化 — コードからシステムへ

同じ「通知」の概念が、スケールによってどう形を変えるかを示します。

```
スケール: コードレベル
────────────────────
  class EventEmitter {
    emit('orderPlaced', event)
  }
  → 同一プロセス内で通知
  → 第4回 デザインパターン（Observer）


スケール: アプリケーションレベル
────────────────────
  class DomainEventPublisher {
    publish(new OrderPlaced(...))
  }
  → ドメインイベントとして意味を持つ
  → 第5回 DDD（ドメインイベント）
  → 第6回 クリーンアーキテクチャ（ユースケースから発行）


スケール: システムレベル
────────────────────
  Kafka/RabbitMQ
  ┌──────┐  OrderPlaced  ┌────────┐
  │Order │ ─────────────▶│Broker  │──▶ InventoryService
  │Svc   │               │        │──▶ PaymentService
  └──────┘               │        │──▶ NotificationService
                         └────────┘
  → サービス間の非同期通信
  → 第9回 イベント駆動アーキテクチャ
```

## 5. 「なぜ」の連鎖 — 概念が必要になる理由のつながり

```
「コードの変更が予期せぬ場所に波及する」
    │
    ▼ だから
「責務を分離し、依存を制御する」 → SOLID原則、カプセル化
    │
    ▼ すると
「繰り返し現れる設計問題に名前がつく」 → デザインパターン
    │
    ▼ さらに
「ビジネスの言葉でコードを書きたい」 → DDD
    │
    ▼ でも
「ビジネスロジックがDB/HTTP/FWに依存している」
    │
    ▼ だから
「レイヤーで依存方向を制御する」 → クリーンアーキテクチャ
    │
    ▼ やがて
「チームが大きくなり、1つのアプリでは開発が衝突する」
    │
    ▼ だから
「サービスを分割して独立にデプロイする」 → マイクロサービス
    │
    ▼ すると
「サービスが増えて手作業の運用が追いつかない」
    │
    ▼ だから
「インフラを自動化し標準化する」 → オーケストレーション
    │
    ▼ 同時に
「同期通信だとサービスが連鎖的に落ちる」
    │
    ▼ だから
「イベントで疎結合に連携する」 → イベント駆動
    │
    ▼ ただし
「結果整合性、冪等性、補償処理の複雑さが増す」
    │
    ▼ だから
「トレードオフを理解し、文脈に応じて判断する」 → 設計判断力
```

## 7. よくある設計判断とその根拠

ワークショップで学ぶ概念を使って、実際の判断がどう行われるかの例です。

```
判断1: 「モノリスのままか、マイクロサービスに分割するか？」
──────────────────────────────────────────────────────

  判断に必要な知識:
  ├── DDD: 境界づけられたコンテキストの識別（第5回）
  ├── マイクロサービス: 分割のメリット・コスト（第7回）
  └── イベント駆動: 分割後の通信方式（第9回）

  判断基準:
  ├── チーム規模が10人以下 → モジュラーモノリスで十分
  ├── スケーリング要件がサービスごとに異なる → 分割を検討
  └── データの結合が強い → 分割すべきではない


判断2: 「同期通信か、非同期通信か？」
──────────────────────────────────────────────────────

  判断に必要な知識:
  ├── マイクロサービス: 通信パターン（第7回）
  ├── イベント駆動: 結果整合性のトレードオフ（第9回）
  └── RDB: ACID vs 結果整合性（第2回、第9回）

  判断基準:
  ├── 即座にレスポンスが必要 → 同期（REST/gRPC）
  ├── 処理の失敗を呼び出し側に伝える必要 → 同期
  ├── 処理に時間がかかる → 非同期（キュー）
  └── 複数のサービスに通知したい → 非同期（イベント）


判断3: 「CQRSを導入するか？」
──────────────────────────────────────────────────────

  判断に必要な知識:
  ├── RDB: 正規化と非正規化のトレードオフ（第2回）
  ├── イベント駆動: CQRS の3レベル（第9回）
  └── DDD: 集約とリードモデルの違い（第5回）

  判断基準:
  ├── 読み書きの負荷が大きく異なる → 導入を検討
  ├── 複数集約をまたぐ複雑な読み取りが多い → 導入を検討
  ├── シンプルなCRUDで十分 → 不要
  └── 結果整合性の複雑さを許容できない → 不要
```
