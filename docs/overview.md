# 企画概要・コンセプト

## 企画名

**アーキテクチャ登竜門** — Architecture Toryumon

## コンセプト

「登竜門」とは、困難だが突破すれば大きく成長できる関門を意味します。本ワークショップでは、ソフトウェアアーキテクチャの各テーマを一つずつ登り、基礎から応用まで体系的に知識を積み上げていきます。

個々の技術を「点」として学ぶのではなく、トピック間のつながりを意識し「線」として理解することで、実務で設計判断ができるエンジニアを育成します。

## 背景・なぜこの企画が必要か

実務経験2〜5年目のエンジニアは、コードを書く力は身についていても、以下の課題に直面することが多くあります。

- **設計判断の根拠が持てない** — なぜこの構造にするのか説明できない
- **知識が断片的** — DDD、クリーンアーキテクチャ等の用語は知っているが体系的に理解していない
- **全体像が見えない** — オブジェクト指向からマイクロサービスまでがどうつながるか分からない
- **実務への適用が難しい** — 書籍で学んでも、現場のコードにどう適用するか判断できない

本ワークショップは、これらの課題を解決するために、**基礎から応用へと段階的に積み上げる構成**で設計しています。

## 目的・ゴール

### 全体目的

ソフトウェアアーキテクチャの基礎〜応用を体系的に学び、**設計判断の引き出し**を増やす。

### 到達目標

本ワークショップを完了した参加者は、以下ができるようになることを目指します。

1. **オブジェクト指向の原則に基づいた設計ができる** — SOLID原則を意識したクラス設計
2. **データベース設計の基本を押さえている** — 正規化、インデックス戦略を理解している
3. **ドメインモデルを設計できる** — ユビキタス言語を用いた概念モデルの構築
4. **適切なデザインパターンを選択できる** — 問題に対してパターンの適用判断ができる
5. **レイヤー構造を設計できる** — クリーンアーキテクチャの原則に沿った構成
6. **分散システムの基本的なトレードオフを理解している** — マイクロサービス、イベント駆動の利点と課題

## 対象者

### 想定する参加者像

- 実務経験 2〜5年目のソフトウェアエンジニア
- 日常的にTypeScript / JavaScript でコードを書いている
- チーム開発の経験がある（Git、コードレビュー等）
- 「もっと良い設計にしたい」と感じているが、具体的にどうすれば良いか分からない

### 前提知識

- TypeScript の基本文法（型、インターフェース、クラス、非同期処理）
- Node.js の基本的な使い方
- Git の基本操作
- SQLの基本（SELECT, INSERT, UPDATE, DELETE）

### 対象外

- プログラミング未経験者
- すでにアーキテクトとして設計経験が豊富な方（レビュアー・メンターとしての参加は歓迎）

## 形式

| 項目 | 内容 |
|------|------|
| 形式 | 座学中心（講義 + コード例 + 議論） |
| 1回あたりの時間 | 2〜2.5時間 |
| 頻度 | 週1回 |
| 全回数 | 全11回（オリエンテーション + 9テーマ + 総合演習） |
| 期間 | 約3ヶ月 |
| 技術スタック | TypeScript / Node.js |
| 課題 | 各回終了後に復習課題を出題 |

### 各回の構成（標準）

```
[5分]  前回の振り返り・課題の共有
[80分] 講義（概念説明 + TypeScriptコード例）
[10分] 休憩
[20分] ディスカッション・Q&A
[5分]  次回予告・課題説明
```

## 運営方針

### 大切にすること

- **「なぜ」を大切にする** — 技術の使い方だけでなく、なぜその設計が必要かを伝える
- **段階的な積み上げ** — 前回までの知識を土台に、新しい概念を積み上げる
- **具体的なコード例** — 抽象的な話だけでなく、TypeScriptの具体例で理解を支援
- **議論を重視** — 一方通行の講義ではなく、参加者同士の対話を促す
- **完璧を目指さない** — 全てを網羅するのではなく、核となる概念の理解を優先する

### 講師に求めること

- 各テーマについて実務経験に基づいた具体例を交えて説明できること
- 参加者の質問に対して、教科書的な回答だけでなく実務的な視点で答えられること
- 「正解」を押し付けるのではなく、トレードオフを示して考えさせるスタンス

## 期待される成果

- 参加者がアーキテクチャについて**共通言語**を持てるようになる
- 設計レビューやアーキテクチャ選定の場で**根拠を持って議論**できるようになる
- チーム全体の**設計力の底上げ**につながる
