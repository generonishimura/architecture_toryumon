## 1. 全体の地図

本ワークショップの概念は、**扱うスケールが段階的に大きくなる**構造を持っています。

```
スケール    扱う範囲                対応する概念
─────────────────────────────────────────────────────────
  小        1つのクラス・関数       オブジェクト指向、SOLID原則
  ↓         1つのテーブル・DB       RDB、データモデリング
  ↓         複数クラスの連携        デザインパターン
  ↓         1つのドメイン           ドメイン駆動設計 (DDD)
  ↓         1つのアプリケーション    クリーンアーキテクチャ
  ↓         複数のサービス          マイクロサービス
  ↓         インフラ・デプロイ      オーケストレーション
  大        サービス間通信          イベント駆動アーキテクチャ
```

各概念は独立した「点」ではなく、**前の概念が次の概念の土台になる「線」** として設計されています。

### なぜこの順番なのか

- SOLID原則を知らなければ、デザインパターンの価値がわからない
- デザインパターンを知らなければ、DDDの実装が想像できない
- DDDの境界づけられたコンテキストを知らなければ、マイクロサービスの分割判断ができない
- クリーンアーキテクチャを知らなければ、各サービスの内部構造が設計できない

この **積み上げの構造** が、本ワークショップ最大の特徴です。

## 2. オブジェクト指向設計

### この概念が解決する問題

> 「コードを修正したら、関係ないはずの場所が壊れた」

これはソフトウェア開発で最も多い悩みの一つです。原因の多くは、**責務の境界が曖昧** で **依存関係が制御されていない** ことにあります。

### 核心：カプセル化とポリモーフィズム

オブジェクト指向の本質は「継承」ではありません。重要なのは次の2つです。

**カプセル化** — 内部の状態を隠し、外部にはインターフェース（操作方法）だけを公開する。

```
┌──────────────────────┐
│  BankAccount          │
│  ─────────────────    │
│  - balance: number    │  ← 外からは直接触れない
│  ─────────────────    │
│  + deposit(amount)    │  ← 公開された操作を通じてのみ変更
│  + withdraw(amount)   │
│  + getBalance()       │
└──────────────────────┘
```

なぜ大事か：内部の実装（balanceをどう管理するか）を変更しても、外部のコードは影響を受けません。

**ポリモーフィズム** — 同じインターフェースで異なる振る舞いを実現する。

```
         ┌───────────────┐
         │  <<interface>> │
         │  Notifier      │
         │  + send()      │
         └───────┬───────┘
           ┌─────┼─────┐
           ▼     ▼     ▼
        Email  Slack   SMS
        Notif  Notif   Notif
```

呼び出す側は「通知を送る」としか知らない。メール・Slack・SMSのどれかは、実行時に決まります。

### SOLID原則 — 良い設計の5つの指針

| 原則 | 一言で言うと | 身近な例え |
|------|------------|-----------|
| **S** — 単一責任 | 1つのクラスに1つの理由で変更 | 「料理人は料理だけ、会計は会計だけ」 |
| **O** — 開放閉鎖 | 既存コードを修正せず拡張できる | 「コンセントに新しい家電を挿すだけ」 |
| **L** — リスコフの置換 | 親クラスの代わりに子クラスを使える | 「アルバイトでも正社員でも同じマニュアルで動ける」 |
| **I** — インターフェース分離 | 使わないメソッドに依存させない | 「リモコンのボタンは必要なものだけ」 |
| **D** — 依存性逆転 | 具体ではなく抽象に依存する | 「コンセントの形状（規格）に合えばどの家電でもOK」 |

特に **依存性逆転の原則（DIP）** は、以降のすべての概念の土台になる最重要原則です。

### 依存性逆転の原則（DIP）— 深堀り

通常の依存：

```
OrderService ──依存──▶ MySQLDatabase
```

この場合、データベースを変更すると OrderService も修正が必要です。

DIPを適用した依存：

```
OrderService ──依存──▶ <<interface>> OrderRepository ◀──実装── MySQLOrderRepository
```

OrderService は「データの保存・取得」という抽象だけに依存し、MySQLかPostgreSQLかは気にしません。これが **依存の方向を逆転** させるということです。

### 後続の概念とのつながり

- **デザインパターン** → GoFパターンはOOPの設計テクニックそのもの
- **DDD** → エンティティ、値オブジェクトはOOPのクラス設計が前提
- **クリーンアーキテクチャ** → DIPがレイヤー間の依存制御の核心

## 3. リレーショナルデータベース

### この概念が解決する問題

> 「アプリは動いているが、データが壊れていた」

ソフトウェアで最も価値があるのは **データ** です。コードはいつでも書き直せますが、壊れたデータは復元できません。RDBは **データの整合性を仕組みで守る** 技術です。

### 正規化 — データの重複を排除する

正規化とは、データの冗長性を排除し、更新時の矛盾を防ぐ手法です。

```
【正規化前】注文テーブル
┌────────┬──────────┬────────────┬──────────┐
│ 注文ID  │ 商品名    │ 顧客名      │ 顧客住所  │
├────────┼──────────┼────────────┼──────────┤
│ 001    │ ノートPC  │ 田中太郎    │ 東京都... │
│ 002    │ マウス    │ 田中太郎    │ 東京都... │  ← 顧客情報が重複！
└────────┴──────────┴────────────┴──────────┘

田中さんが引っ越したら、すべての行を更新する必要がある
→ 1行でも更新漏れがあるとデータ不整合

【正規化後】
注文テーブル                 顧客テーブル
┌────────┬──────┬───────┐   ┌───────┬──────────┬──────────┐
│ 注文ID  │ 商品ID│ 顧客ID │   │ 顧客ID │ 顧客名    │ 顧客住所  │
├────────┼──────┼───────┤   ├───────┼──────────┼──────────┤
│ 001    │ P01  │ C01   │   │ C01   │ 田中太郎  │ 東京都... │
│ 002    │ P02  │ C01   │   └───────┴──────────┴──────────┘
└────────┴──────┴───────┘
```

| レベル | 要件 | 平たく言うと |
|--------|------|------------|
| 第1正規形 | 繰り返し項目の排除、値の原子性 | 「1つのセルに1つの値」 |
| 第2正規形 | 部分関数従属の排除 | 「主キーの一部だけに依存する列を分離」 |
| 第3正規形 | 推移的関数従属の排除 | 「主キー以外の列に依存する列を分離」 |

### インデックス — 検索を高速化する仕組み

インデックスは **本の索引** と同じです。全ページを読まなくても目的の情報にたどり着けます。

```
インデックスなし: テーブル全行を順番にスキャン → O(n)
インデックスあり: B-Treeで効率的に検索      → O(log n)

100万行のテーブル:
  全行スキャン → 100万回の比較
  B-Tree検索   → 約20回の比較
```

**注意点：** インデックスは読み取りを高速化する一方、書き込み時にはインデックスの更新コストがかかります。すべての列にインデックスを貼るのではなく、**よく検索される列** に限定することが重要です。

### トランザクションとACID特性

トランザクションは **「一連の操作を全部成功させるか、全部なかったことにするか」** を保証する仕組みです。

```
銀行の送金処理:
  1. Aさんの口座から1万円引く
  2. Bさんの口座に1万円足す

→ 1だけ成功して2が失敗したら、1万円が消える！
→ トランザクションなら、2が失敗した時点で1も取り消される
```

| 特性 | 意味 | 例え |
|------|------|------|
| **A** — 原子性 | 全部成功 or 全部失敗 | 「途中で止まらない」 |
| **C** — 整合性 | 制約が常に満たされる | 「残高はマイナスにならない」 |
| **I** — 分離性 | 他のトランザクションの影響を制御 | 「他の人の処理中を見えなくする」 |
| **D** — 永続性 | コミットしたデータは消えない | 「確定した取引は記録される」 |

### 後続の概念とのつながり

- **データモデリング** → 正規化の知識が概念モデルから物理モデルへの変換に必須
- **DDD** → リポジトリパターンでドメインモデルとDBの橋渡し
- **マイクロサービス** → Database per Service パターンの設計判断に直結

## 4. データモデリング

### この概念が解決する問題

> 「テーブル設計から始めたが、ビジネスの要件変更のたびに大改修が必要になる」

テーブル設計（物理モデル）からいきなり始めると、**ビジネスの構造** ではなく **技術の制約** に引きずられた設計になります。データモデリングは、ビジネスを正確に構造化してから技術に落とし込む手法です。

### 3段階のモデリング

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  概念モデル    │ ──▶ │  論理モデル    │ ──▶ │  物理モデル    │
│              │     │              │     │              │
│ ビジネスの    │     │ データ構造の  │     │ DB固有の      │
│ エンティティと │     │ 正規化・型    │     │ 型・制約・     │
│ 関係を整理    │     │ の決定       │     │ インデックス    │
│              │     │              │     │              │
│ 技術を       │     │ 特定のDBに   │     │ 実際のDB    │
│ 考えない      │     │ 依存しない    │     │ に合わせる    │
└──────────────┘     └──────────────┘     └──────────────┘
  WHO / WHAT            HOW (論理)          HOW (実装)
```

**なぜ3段階か？**

- 概念モデルで **何が重要か** を明確にする（ビジネスとの認識合わせ）
- 論理モデルで **データの関係** を厳密にする（正規化の適用）
- 物理モデルで **パフォーマンス** を考慮する（インデックス、非正規化の判断）

いきなり物理モデルを書くと、「このテーブルは何のために存在するのか」が曖昧になり、後々の変更が困難になります。

### ER図の読み方

```
  ┌──────────┐    1    N    ┌──────────┐
  │  Author   │────────────│  Book     │
  │  著者     │            │  書籍     │
  └──────────┘             └──────────┘

  「1人の著者は N冊の本を書く。1冊の本は1人の著者が書く」

  多対多の場合 → 中間テーブルで分解

  ┌──────────┐    N    N    ┌──────────┐
  │  Student  │────────────│  Course   │
  │  学生     │            │  講座     │
  └──────────┘             └──────────┘
                  ↓ 分解
  Student  ──1:N──  Enrollment  ──N:1──  Course
                    （受講登録）
```

### 後続の概念とのつながり

- **DDD** → 概念モデルはドメインモデルの出発点。エンティティと値オブジェクトの区別は概念モデリングで養われる
- **マイクロサービス** → サービスごとのデータ分割は、概念モデルでのエンティティの境界から導かれる

## 5. デザインパターン

### この概念が解決する問題

> 「設計レビューで指摘されるが、どう直せばいいかわからない」

デザインパターンは、ソフトウェア設計で **繰り返し現れる問題** に対する **再利用可能な解決策** に名前をつけたものです。パターンを知ることで、問題を見抜く力と、解決策を説明する語彙が得られます。

### パターンの3つのカテゴリ

```
┌──────────────────────────────────────────────────────────┐
│                     デザインパターン                        │
│                                                          │
│  ┌────────────┐  ┌────────────┐  ┌────────────────────┐  │
│  │ 生成パターン │  │ 構造パターン │  │ 振る舞いパターン     │  │
│  │            │  │            │  │                    │  │
│  │ オブジェクト │  │ オブジェクト │  │ オブジェクト間の     │  │
│  │ の作り方    │  │ の組み合せ方│  │ 責務の分担          │  │
│  │            │  │            │  │                    │  │
│  │ • Factory  │  │ • Adapter  │  │ • Strategy         │  │
│  │ • Builder  │  │ • Facade   │  │ • Observer         │  │
│  │            │  │            │  │ • Repository       │  │
│  └────────────┘  └────────────┘  └────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

### 重要パターンの直感的理解

**Strategy（戦略）パターン** — アルゴリズムを差し替え可能にする

```
問題: if文の分岐が増え続ける
  if (type === "regular") { ... }
  else if (type === "member") { ... }
  else if (type === "vip") { ... }    ← 新しい種別を追加するたびに修正

解決: 戦略を別クラスに分離
  PriceCalculator → PricingStrategy(interface)
                        ├── RegularPricing
                        ├── MemberPricing
                        └── VIPPricing    ← 追加しても既存コードは変更不要
```

**Repository（リポジトリ）パターン** — データアクセスを抽象化する

```
問題: ビジネスロジックの中にSQL文が埋め込まれている
  → DBを変更すると、ビジネスロジックも修正が必要

解決: データアクセスをインターフェースの裏に隠す
  UseCase → Repository(interface) ← InMemoryRepo（テスト用）
                                  ← PrismaRepo（本番用）
                                  ← MockRepo（開発用）
```

**Observer（観察者）パターン** — 状態変化を通知する

```
問題: 注文完了時に「メール送信」「在庫更新」「ログ記録」を実行したい
  → OrderServiceに全部書くと、責務が肥大化する

解決: 「注文完了」というイベントを発行し、各処理が個別に購読
  OrderService ──発行──▶ "OrderCompleted"
                              │
                   ┌──────────┼──────────┐
                   ▼          ▼          ▼
              EmailSender  Inventory  Logger
```

### パターン適用の心得

**「問題が先、パターンは後」** — パターンを使うこと自体が目的になってはいけません。

```
✗ 「Factoryパターンを使おう」→ 適用先を探す
✓ 「オブジェクト生成のロジックが複雑すぎる」→ Factory が適切か検討する
```

### 後続の概念とのつながり

- **DDD** → Repository, Factory, Strategy が直接活用される
- **クリーンアーキテクチャ** → Adapter, Gateway で外部との境界を設計
- **イベント駆動** → Observer パターンがシステム規模に拡張される

## 6. ドメイン駆動設計

### この概念が解決する問題

> 「コードとビジネスの話が噛み合わない。開発者はテーブル名で話し、ビジネス側は業務用語で話す」

DDDは **ビジネスの構造をそのままコードに写し取る** ための設計手法です。技術的なテクニックではなく、**ビジネスへのアプローチ** です。

### 2つの柱：戦略的設計と戦術的設計

```
┌──────────────────────────────────────────────────────┐
│              ドメイン駆動設計 (DDD)                     │
│                                                      │
│  ┌─────────────────────┐  ┌─────────────────────┐   │
│  │   戦略的設計          │  │   戦術的設計          │   │
│  │   (大きな構造)        │  │   (コードレベル)      │   │
│  │                     │  │                     │   │
│  │  • ユビキタス言語     │  │  • エンティティ       │   │
│  │  • 境界づけられた     │  │  • 値オブジェクト     │   │
│  │    コンテキスト       │  │  • 集約              │   │
│  │  • コンテキストマップ  │  │  • ドメインサービス    │   │
│  │                     │  │  • ドメインイベント    │   │
│  │                     │  │  • リポジトリ         │   │
│  └─────────────────────┘  └─────────────────────┘   │
│                                                      │
│     「どこで分けるか」         「中をどう作るか」        │
└──────────────────────────────────────────────────────┘
```

### ユビキタス言語 — 全員が同じ言葉を使う

```
✗ 開発者：「user_order テーブルの status カラムを 2 にする」
✓ 全員  ：「注文を確定する」

コードにも同じ言葉を使う:
  order.confirm()  ← 「注文を確定する」がそのままメソッド名
```

### 境界づけられたコンテキスト — 同じ言葉が異なる意味を持つ場所

```
「商品」という同じ言葉でも、文脈によって意味が異なる:

┌─ 商品カタログ ──────┐  ┌─ 在庫管理 ──────────┐  ┌─ 配送 ────────────┐
│                    │  │                    │  │                  │
│ 商品 =             │  │ 商品 =             │  │ 商品 =           │
│  名前、説明文、     │  │  SKU、倉庫、        │  │  重量、サイズ、    │
│  画像、カテゴリ     │  │  在庫数、入荷日     │  │  配送方法         │
│                    │  │                    │  │                  │
└────────────────────┘  └────────────────────┘  └──────────────────┘

→ それぞれが独立した「境界づけられたコンテキスト」
→ 後にマイクロサービスの分割単位になる
```

### エンティティと値オブジェクト — 「誰であるか」と「何であるか」

```
エンティティ（同一性で識別）         値オブジェクト（属性の組み合わせで識別）
┌───────────────────────┐        ┌───────────────────────┐
│ 注文 (Order)           │        │ 金額 (Money)           │
│ ───────────────        │        │ ───────────────        │
│ id: "ORD-001"          │        │ amount: 1000           │
│ → idが同じなら同じ注文   │        │ currency: "JPY"        │
│ → 内容が変わっても      │        │ → 1000円は1000円        │
│   同じ注文              │        │ → 区別する必要がない     │
│                        │        │ → 不変（イミュータブル） │
└───────────────────────┘        └───────────────────────┘

判断基準: 「名前を変えたら別物になるか？」
  人 → 名前を変えても同一人物 → エンティティ
  住所 → 同じ住所なら区別不要  → 値オブジェクト
```

### 集約 — データの整合性を守る境界

```
┌─ 注文(集約) ─────────────────────────────────────┐
│                                                  │
│  ┌──────────────────┐                            │
│  │ Order（集約ルート） │ ← 外部からのアクセスはここだけ │
│  │  id, status       │                            │
│  │  confirm()        │                            │
│  │  addItem()        │                            │
│  └──────┬───────────┘                            │
│         │ 内包                                    │
│  ┌──────┴───────────┐                            │
│  │ OrderItem         │ ← 外部から直接操作しない      │
│  │  productId        │                            │
│  │  quantity         │                            │
│  │  unitPrice        │                            │
│  └──────────────────┘                            │
│                                                  │
└──────────────────────────────────────────────────┘

ルール: 集約の外から内部のオブジェクトを直接変更してはいけない
  ✗ order.items[0].quantity = 5
  ✓ order.changeItemQuantity(productId, 5)
```

### ドメインイベント — 「何が起きたか」を記録する

```
命令（コマンド）:   「注文を確定せよ」     → order.confirm()
事実（イベント）:   「注文が確定された」   → OrderPlaced { orderId, timestamp }

イベントの特徴:
  • 過去形で命名する（OrderPlaced, PaymentReceived）
  • 事実なので取り消せない
  • 他のシステムに伝播させることができる
```

### 後続の概念とのつながり

- **クリーンアーキテクチャ** → DDDのドメインモデルがドメイン層に配置される
- **マイクロサービス** → 境界づけられたコンテキストがサービス分割の指針になる
- **イベント駆動** → ドメインイベントがサービス間通信の基盤になる

## 7. クリーンアーキテクチャ

### この概念が解決する問題

> 「フレームワークのバージョンアップでビジネスロジックまで書き直しになった」

クリーンアーキテクチャの目的は **ビジネスロジックを外部の技術的詳細から守る** ことです。フレームワーク、データベース、UIは「詳細」であり、ビジネスルールこそが「核」です。

### 同心円モデルと依存ルール

```
                ┌─────────────────────────────────────┐
                │  Frameworks & Drivers（外部層）        │
                │  Express, Prisma, 外部API              │
                │    ┌──────────────────────────────┐  │
                │    │  Interface Adapters（変換層）   │  │
                │    │  Controller, Presenter        │  │
                │    │    ┌──────────────────────┐  │  │
                │    │    │  Use Cases（応用層）   │  │  │
                │    │    │  アプリ固有のルール    │  │  │
                │    │    │    ┌──────────────┐  │  │  │
                │    │    │    │  Entities     │  │  │  │
                │    │    │    │  （ドメイン層）│  │  │  │
                │    │    │    │  ビジネスの   │  │  │  │
                │    │    │    │  核心ルール   │  │  │  │
                │    │    │    └──────────────┘  │  │  │
                │    │    └──────────────────────┘  │  │
                │    └──────────────────────────────┘  │
                └─────────────────────────────────────┘

依存ルール: 矢印は必ず内側に向かう（外 → 内）
  • 内側の層は外側の層を知らない
  • ドメイン層は Express も Prisma も知らない
  • ユースケース層は HTTP リクエストの形を知らない
```

### 各層の役割

| 層 | 責務 | 含まれるもの | 知らないこと |
|---|------|------------|------------|
| **Entities** | ビジネスルール | エンティティ、値オブジェクト、リポジトリインターフェース | HTTP, DB, フレームワーク |
| **Use Cases** | アプリのユースケース | 「注文を作成する」等の操作 | HTTPリクエストの形、DBの種類 |
| **Interface Adapters** | データ形式の変換 | Controller, Presenter, Gateway | 具体的なDB実装 |
| **Frameworks** | 技術的詳細 | Express設定, Prisma設定 | — |

### 依存性逆転が実現する「交換可能性」

```
テスト時:                        本番時:
  UseCase                         UseCase
    │                                │
    ▼                                ▼
  OrderRepository(interface)       OrderRepository(interface)
    │                                │
    ▼                                ▼
  InMemoryOrderRepository         PrismaOrderRepository
  （メモリ上で高速テスト）           （実際のDB操作）
```

ドメイン層にインターフェースを置き、インフラ層で実装する。これにより：
- テスト時はDBなしで高速に実行
- 本番DBの変更がビジネスロジックに影響しない
- フレームワークの変更が最小限の影響で済む

### ディレクトリ構成例

```
src/
├── domain/              # 内側: ビジネスの核心
│   ├── entities/        #   Order, Product, User
│   ├── value-objects/   #   Money, Address, Email
│   └── repositories/    #   OrderRepository(interface)
│
├── application/         # ユースケース
│   ├── use-cases/       #   PlaceOrderUseCase, CancelOrderUseCase
│   └── dto/             #   Input/Output の型定義
│
├── infrastructure/      # 外側: 技術的詳細
│   ├── persistence/     #   PrismaOrderRepository
│   ├── external/        #   外部API連携
│   └── config/          #   環境設定
│
└── presentation/        # 外側: UIとの接続
    ├── controllers/     #   OrderController
    ├── routes/          #   Express ルーティング
    └── middleware/       #   認証、エラーハンドリング
```

### 後続の概念とのつながり

- **マイクロサービス** → 各サービスの内部構造としてクリーンアーキテクチャを適用
- **イベント駆動** → ユースケース層がドメインイベントを発行し、インフラ層で配信

## 8. マイクロサービス

### この概念が解決する問題

> 「チームが10人を超えたあたりから、お互いのコード変更が衝突するようになった」

マイクロサービスは、**独立してデプロイ可能な小さなサービス** にシステムを分割するアーキテクチャスタイルです。しかし **分割にはコストが伴う** ことを理解することが最も重要です。

### モノリスから分割へ — 段階的なアプローチ

```
段階1: モノリス            段階2: モジュラーモノリス    段階3: マイクロサービス
┌───────────────┐        ┌───────────────┐         ┌─────┐ ┌─────┐ ┌─────┐
│ すべてが1つの   │        │ 1つのデプロイ   │         │注文  │ │在庫  │ │決済  │
│ アプリに同居    │   →    │ だが内部は     │   →     │サービス│ │サービス│ │サービス│
│               │        │ モジュールで    │         │     │ │     │ │     │
│               │        │ 分離されている  │         │ DB  │ │ DB  │ │ DB  │
│     DB        │        │     DB        │         └─────┘ └─────┘ └─────┘
└───────────────┘        └───────────────┘

                  最初から分割しない！
                  まずモノリスで始めて、必要に応じて分割する
```

### サービス分割の判断基準

DDDの **境界づけられたコンテキスト** がサービスの分割単位になります。

```
分割すべきサイン:                    分割すべきでないサイン:
✓ チームが独立して開発したい          ✗ 「マイクロサービスが流行っているから」
✓ 異なるスケーリング要件がある        ✗ データを頻繁にJOINする必要がある
✓ 異なる技術スタックが最適           ✗ 分散トランザクションが必要になる
✓ 障害の影響範囲を限定したい          ✗ チームが小さい（2-3人）
```

### サービス間通信

```
同期通信                             非同期通信
┌──────┐  HTTP/gRPC  ┌──────┐      ┌──────┐  イベント  ┌───────┐  ┌──────┐
│注文   │ ──────────▶ │在庫   │      │注文   │ ────────▶│Broker │──▶│在庫   │
│サービス│ ◀────────── │サービス│      │サービス│          └───────┘  │サービス│
└──────┘   レスポンス  └──────┘      └──────┘                     └──────┘

同期: シンプル、即座に結果がわかる        非同期: 疎結合、障害に強い
     相手が落ちていると失敗する               複雑、結果整合性の考慮が必要
```

### Database per Service — 各サービスが自分のDBを持つ

```
✗ 共有データベース（アンチパターン）
┌──────┐  ┌──────┐
│注文   │  │在庫   │
│サービス│  │サービス│
└──┬───┘  └──┬───┘
   │         │
   └────┬────┘
        ▼
   ┌─────────┐
   │ 共有 DB  │  ← テーブルの変更が全サービスに影響
   └─────────┘     独立デプロイが不可能になる

✓ Database per Service
┌──────┐  ┌──────┐
│注文   │  │在庫   │
│サービス│  │サービス│
└──┬───┘  └──┬───┘
   ▼         ▼
┌─────┐  ┌─────┐
│注文DB│  │在庫DB│  ← 各サービスが独自のDBを所有
└─────┘  └─────┘     他のサービスのデータはAPIを通じて取得
```

### 耐障害性パターン — Circuit Breaker

```
         状態遷移図:

                    成功
         ┌─────────────────────┐
         ▼                     │
    ┌─────────┐  失敗が閾値超過  ┌──────┐  タイムアウト後  ┌───────────┐
    │ CLOSED  │ ──────────────▶ │ OPEN │ ─────────────▶ │ HALF-OPEN │
    │ (通常)  │                 │(遮断) │                │ (試行)     │
    └─────────┘                 └──────┘                └───────────┘
                                  ▲                         │
                                  │        失敗             │
                                  └─────────────────────────┘

CLOSED:  通常通りリクエストを通す
OPEN:    リクエストを即座に失敗させる（相手に負荷をかけない）
HALF-OPEN: 1つだけ試しに通して、成功すればCLOSEDに戻す
```

### 後続の概念とのつながり

- **オーケストレーション** → サービスのデプロイ・スケーリング・管理の自動化
- **イベント駆動** → サービス間の非同期通信と結果整合性の実現

## 9. オーケストレーション

### この概念が解決する問題

> 「サービスが10個に増えたが、デプロイに半日かかる。障害時にどのサービスが原因か分からない」

マイクロサービスの数が増えると、手作業での運用は限界を迎えます。オーケストレーションは **インフラの自動化と標準化** でこの問題を解決します。

### コンテナ技術 — アプリケーションを箱に詰める

```
従来の環境:                     コンテナ:
┌──────────────────┐           ┌─────────┐ ┌─────────┐ ┌─────────┐
│ サーバー          │           │ App A   │ │ App B   │ │ App C   │
│ ┌──┐ ┌──┐ ┌──┐  │           │ + 依存  │ │ + 依存  │ │ + 依存  │
│ │A │ │B │ │C │  │           └────┬────┘ └────┬────┘ └────┬────┘
│ └──┘ └──┘ └──┘  │                │          │          │
│  依存ライブラリが  │           ┌────┴──────────┴──────────┴────┐
│  衝突する可能性    │           │      コンテナランタイム         │
│                  │           │      （Docker）               │
│  OS              │           │      OS                      │
└──────────────────┘           └───────────────────────────────┘

各アプリが独立した環境を持つ → 依存の衝突がなくなる
「自分のPCでは動く」問題が解消される
```

### Kubernetes — コンテナの管理者

```
Kubernetesの主要リソース:

Pod（最小単位）
  └── Container（アプリ）
       1つ以上のコンテナをまとめた単位

Deployment（レプリカ管理）
  └── Pod × N個（レプリカ数を宣言的に管理）
       「常に3つのPodを維持せよ」→ 1つ落ちたら自動で再作成

Service（ネットワーク）
  └── 複数のPodへのトラフィック分散
       「order-service」という名前でアクセスすると
       背後の3つのPodに負荷分散

Ingress（外部公開）
  └── 外部からのHTTPトラフィックをServiceに振り分け
       api.example.com/orders → order-service
       api.example.com/users  → user-service
```

### デプロイ戦略

```
ローリングアップデート:
  v1 v1 v1 → v2 v1 v1 → v2 v2 v1 → v2 v2 v2
  段階的に新バージョンに置き換え。ダウンタイムなし。

ブルーグリーン:
  [Blue: v1] ← トラフィック
  [Green: v2] ← 待機
  切り替え時にトラフィックをGreenに向ける。即座にロールバック可能。

カナリアリリース:
  v1 ← 90%のトラフィック
  v2 ← 10%のトラフィック
  問題がなければ徐々にv2の割合を増やす。リスクを最小化。
```

### CI/CD — 変更を安全に素早く届ける

```
コード変更
    │
    ▼
┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
│  Build  │──▶│  Test  │──▶│  Push  │──▶│ Deploy │
│  構築   │   │ テスト  │   │ イメージ│   │ 配備   │
│        │   │        │   │ 登録   │   │        │
└────────┘   └────────┘   └────────┘   └────────┘
  CI（継続的インテグレーション）    CD（継続的デリバリー）

  自動化により:
  • 人的ミスが減る
  • デプロイ頻度が上がる
  • 問題の早期発見ができる
```

## 10. イベント駆動アーキテクチャ

### この概念が解決する問題

> 「注文サービスが在庫サービスを呼び、在庫が決済を呼び...。1つのサービスが落ちると連鎖的に全体が止まる」

イベント駆動は **サービス間を疎結合にし、非同期で連携する** アーキテクチャスタイルです。

### コマンドとイベントの違い

```
コマンド（命令）:                   イベント（事実）:
「注文を確定せよ」                  「注文が確定された」
 → 特定の相手に送る                 → 不特定多数に通知
 → 失敗する可能性がある              → すでに起きた事実
 → 命令形で命名                     → 過去形で命名
   ConfirmOrder                      OrderConfirmed
   ProcessPayment                    PaymentProcessed
   ShipOrder                         OrderShipped
```

### メッセージブローカー

```
キュー型（RabbitMQ, SQS）:          ストリーム型（Kafka）:
1つのメッセージ → 1つのコンシューマ    1つのイベント → 複数のコンシューマ

Producer → [Queue] → Consumer       Producer → [Topic] → Consumer A
                                                      → Consumer B
                                                      → Consumer C

処理したら消える                      永続的に保存、再読み込み可能
タスク分散向き                        イベント伝播向き
```

### CQRS — 読み書きの責務を分離する

```
従来:                              CQRS:
 ┌─────────────┐                    ┌──────────┐    ┌──────────┐
 │  同じモデルで │                    │ Write    │    │ Read     │
 │  読み書き    │                    │ Model    │    │ Model    │
 │             │                    │          │    │          │
 │  Order      │                    │ Order    │    │ OrderView│
 │  (正規化)   │                    │ (正規化) │    │ (非正規化)│
 └──────┬──────┘                    └────┬─────┘    └────┬─────┘
        │                               │  イベント     │
     ┌──┴──┐                          ┌──┴──┐       ┌──┴──┐
     │ DB  │                          │Write│       │Read │
     └─────┘                          │ DB  │       │ DB  │
                                      └─────┘       └─────┘

なぜ分離するか:
  • 読み取りは高速な非正規化データが最適
  • 書き込みは整合性重視の正規化データが最適
  • 読み書きの負荷が大きく異なる場合に有効
```

### 結果整合性 — 「いつか整合する」という考え方

```
強い整合性（従来のDB）:              結果整合性（分散システム）:
  書き込み → 即座に全員が最新を見る     書き込み → しばらくは古いデータが見える
  シンプルだが、分散環境では困難           → 数秒〜数分で全体が整合する

例: SNSの「いいね」数
  ユーザーAが「いいね」 → ユーザーBが見る: まだ反映されていない
                       → 数秒後: 反映される
  → これで十分なケースは多い
```

### Sagaパターン — 分散トランザクションの代替

```
従来のトランザクション:                Sagaパターン:
  BEGIN                               各サービスがローカルトランザクションを実行
    注文作成                            失敗時は補償トランザクションで「取り消し」
    在庫減少
    決済処理
  COMMIT or ROLLBACK                   ← 分散環境ではこれが使えない

Choreography（振り付け）:
  注文作成 ──▶ OrderPlaced ──▶ 在庫確保 ──▶ InventoryReserved ──▶ 決済処理
                                                                    │
  在庫復元 ◀── PaymentFailed ◀──────────────────────────────────────┘
                (補償トランザクション)

Orchestration（指揮）:
  ┌────────────────────┐
  │ Saga Orchestrator   │
  │  1. 注文作成 → 成功 │
  │  2. 在庫確保 → 成功 │
  │  3. 決済処理 → 失敗 │
  │  → 2を補償（在庫復元）│
  │  → 1を補償（注文取消）│
  └────────────────────┘
```

### 冪等性 — 同じ操作を何度実行しても結果が同じ

```
ネットワーク障害で同じメッセージが2回届く可能性がある:

✗ 非冪等: processPayment(orderId, 1000円)
  → 2回実行すると2000円引かれる！

✓ 冪等: processPayment(eventId: "evt-123", orderId, 1000円)
  → eventIdで処理済みかチェック
  → 2回目は「すでに処理済み」としてスキップ
```

## 11. 概念のスケール感

すべての概念を **スケール（扱う範囲の大きさ）** で整理すると、全体の位置づけが明確になります。

```
                        コード      アプリ      システム
                        レベル      レベル      レベル
                        ─────      ─────      ──────
SOLID原則               ████
カプセル化              ████
ポリモーフィズム          ████

正規化                  ████
ACID / トランザクション   ████
インデックス             ████

データモデリング          ████ ██████

デザインパターン          ████ ██████

DDD（戦術的設計）         ████ ██████
DDD（戦略的設計）              ██████ ██████

クリーンアーキテクチャ          ██████

マイクロサービス                ██████ ██████
オーケストレーション                   ██████
イベント駆動                          ██████

共通する原則:
  ┌──────────────────────────────────────────────┐
  │  • 関心の分離（責務を明確に分ける）              │
  │  • 疎結合（変更の影響を局所化する）              │
  │  • 抽象化（詳細を隠し、本質を明示する）          │
  │  • トレードオフ（銀の弾丸はない）               │
  └──────────────────────────────────────────────┘
```

### 概念間の「同じ原則の異なるスケール」

多くの概念は **同じ原則をスケールを変えて適用したもの** です。

| 原則 | コードレベル | アプリレベル | システムレベル |
|------|------------|------------|--------------|
| 責務の分離 | 単一責任の原則 | レイヤー分離 | サービス分割 |
| 依存の制御 | 依存性逆転の原則 | ポート＆アダプター | API Gateway |
| 変更の局所化 | カプセル化 | 境界づけられたコンテキスト | Database per Service |
| 通知と連携 | Observerパターン | ドメインイベント | イベント駆動アーキテクチャ |
| データアクセスの抽象化 | Repositoryパターン | インフラ層の分離 | API Composition |

### 最も大切なこと

> **「正解は1つではない。トレードオフを理解し、根拠を持って判断できること」**

アーキテクチャに唯一の正解はありません。重要なのは：

1. **選択肢を知っていること** — 引き出しの多さ
2. **トレードオフを説明できること** — 「Xを得る代わりにYを犠牲にする」
3. **文脈に応じて判断できること** — チーム規模、要件、制約を考慮する
4. **判断の根拠を言語化できること** — チームに共有し、議論できる

本ワークショップは、この **「設計判断力」** を鍛えるためのプログラムです。
