# 改善提案まとめ

> 本ドキュメントは、アーキテクチャ登竜門の今後の進め方について、改善点・参加者がつまずきやすいポイント・追加トピックの提案をまとめたものです。

---

## 1. 参加者がつまずきやすいポイントと対策

### 1-1. フェーズ間の「難易度の壁」

カリキュラムの3フェーズ構成は優れているが、特に以下の遷移で参加者が壁を感じやすい。

| 遷移 | 難しさの原因 | 対策案 |
|------|-------------|--------|
| 第3回（データモデリング）→ 第4回（デザインパターン） | データ中心の思考からオブジェクト設計への切り替え | フェーズ2冒頭に「なぜ設計パターンが必要になるのか」をデータモデリングの課題から導入 |
| 第4回（デザインパターン）→ 第5回（DDD） | 技術パターンから戦略的設計への抽象度上昇 | DDDの前に「前提知識チェックリスト」を事前配布。オブジェクト指向・データモデリングの知識がDDDでどう使われるかを明示 |
| 第6回（クリーンアーキテクチャ）→ 第7回（マイクロサービス） | 単一アプリ内の設計から分散システムへのジャンプ | 「モノリスの限界」を具体的に体験するシナリオを用意し、なぜサービス分割が必要になるかを動機づける |

**具体的なアクション:**
- 各フェーズの最初のセッションに **ブリッジセクション（15分）** を追加
- 「前のフェーズで学んだXが、このフェーズではYとして使われる」を明示するスライドを用意

### 1-2. 抽象概念と実務の距離感

DDDの「ユビキタス言語」や「境界づけられたコンテキスト」、クリーンアーキテクチャの「依存性逆転」など、概念としては理解できても「明日の仕事でどう使うか」がイメージしにくい。

**対策案:**
- 各ワークショップに **「実務適用シナリオ」セクション** を追加
  - 「あなたのチームのコードベースにこの概念を適用するとしたら？」を考える時間を設ける
  - Before/After形式で、実際にありがちなコードのリファクタリング例を提示
- `workshops/XX-*/practical-scenarios.md` として、実務寄りのケーススタディを別ファイルで提供

### 1-3. コード例が「見るだけ」になっている

現状のTypeScriptコード例はMarkdown内の提示のみで、参加者が手を動かす機会が少ない。

**対策案:**
- `exercises/` ディレクトリを新設し、各回のコード例を **実行可能なTypeScriptプロジェクト** として提供
- `package.json` + `tsconfig.json` を整備し、以下のように実行できる環境を構築:

```bash
npm install
npm run exercise:01  # 第1回の演習を実行
npm run exercise:02  # 第2回の演習を実行
```

- 各演習にテストを付けて、参加者が自分の理解を検証できるようにする
- 例: 「このクラスをSOLID原則に沿ってリファクタリングせよ。テストが全て通ればOK」

### 1-4. 課題の模範解答がない

`schedule.md` で各回の課題出題は言及されているが、模範解答や解説が存在しない。参加者が自学自習する際の指針がない。

**対策案:**
- 各ワークショップに `homework-solutions.md` を追加
- 講師が課題のフィードバックを行った後に参加者に共有する想定
- 「正解は1つではない」ことを示すため、複数のアプローチとそのトレードオフを提示

## 2. カリキュラム構造の改善

### 2-1. 中間振り返りの構造化

`schedule.md` にWeek5の中間振り返り（任意）とあるが、内容が未定義。

**提案: `workshops/mid-review/` を新設**

内容:
- フェーズ1の理解度セルフチェック（10問程度）
- フェーズ1の概念がフェーズ2でどう使われるかのプレビュー
- 参加者同士のペアレビュー（課題の振り返り）
- ここまでの疑問点の共有と解消

### 2-2. 学習成果の可視化

参加者の成長を測る仕組みが弱い（KPTは最終回のみ）。

**提案: 設計力セルフアセスメント**

- オリエンテーションで各テーマの自信度を1-5で評価（事前）
- 最終回で同じアセスメントを再実施（事後）
- 事前・事後の差分を可視化し、成長を実感できるようにする
- `docs/assessment-template.md` としてテンプレートを提供（後述）

### 2-3. 一貫したサンプルプロジェクトの導入

現状、各回のコード例が独立しており、回をまたいだ学びのつながりが弱い。

**提案: 全回を通して同じ題材を使う**

例えば「タスク管理アプリ」を題材に:

| 回 | 適用する学び |
|----|------------|
| 第1回 | Taskクラスの設計（SOLID原則） |
| 第2回 | tasksテーブルの設計（正規化、インデックス） |
| 第3回 | タスク管理ドメインの概念/論理/物理モデル |
| 第4回 | Strategy（優先度判定）、Observer（通知）の適用 |
| 第5回 | タスク管理の境界づけられたコンテキスト、集約設計 |
| 第6回 | レイヤー分離（ドメイン層 / ユースケース層 / インフラ層） |
| 第7回 | タスクサービスと通知サービスの分割検討 |
| 第8回 | Docker化、CI/CDパイプライン |
| 第9回 | タスク完了イベント → 通知・集計への非同期処理 |

これにより **「点」の学びが「線」でつながる** というコンセプトが体験として実現できる。

## 3. プラスアルファ：開発手法・追加トピック

### 3-1. テスト戦略（強く推奨）

**なぜ必要か:** アーキテクチャとテストは表裏一体。クリーンアーキテクチャの価値のひとつ「テスタビリティ」は、テスト戦略なしには実感できない。

**カバーすべき内容:**
- テストピラミッド（単体テスト / 統合テスト / E2Eテスト）
- 依存性注入とモックの関係（なぜレイヤー分離がテストを容易にするか）
- テストダブルの使い分け（Mock / Stub / Spy / Fake）
- TDDの基本サイクル（Red → Green → Refactor）

**追加方法の選択肢:**
1. 第6回（クリーンアーキテクチャ）にテスト戦略セクション（30分）を組み込む
2. 独立した回として追加（全12回に拡張）

### 3-2. アジャイル開発手法とアーキテクチャの関係（推奨）

**なぜ必要か:** 参加者が「最初から完璧なアーキテクチャを設計しなければならない」と誤解するリスクがある。アーキテクチャは段階的に進化させるものであり、アジャイル開発との親和性を理解することが重要。

**カバーすべき内容:**
- Evolutionary Architecture（進化的アーキテクチャ）— アーキテクチャは変わるもの
- YAGNI原則 — 必要になるまで作らない
- ADR（Architecture Decision Records）— 設計判断の記録
- 技術的負債の管理 — 意図的な負債と無意識の負債
- フィーチャーフラグ、ストラングラーパターン — 段階的な移行手法

**追加方法の選択肢:**
1. オリエンテーション（第0回）に「学び方の前提」として軽く導入
2. 最終回（第10回）に「実務への適用」セクションとして追加
3. `docs/development-methodology-guide.md` として補足資料化（後述で作成）

### 3-3. 可観測性（Observability）（推奨）

**なぜ必要か:** 分散システム（フェーズ3）では、システムの内部状態を外部から把握できることが運用の前提。設計段階から可観測性を意識する必要がある。

**カバーすべき内容:**
- 可観測性の3本柱: ログ / メトリクス / 分散トレーシング
- OpenTelemetry の概要
- SLI / SLO / エラーバジェットの考え方
- 構造化ログとコリレーションID

**追加方法:** 第8回（オーケストレーション）に20分程度のセクションとして組み込み

### 3-4. セキュリティ設計の基礎（検討）

**なぜ必要か:** 第10回で「扱わなかったテーマ」として言及されているが、認証・認可はほぼすべてのシステムで必須。

**カバーすべき内容:**
- 認証（Authentication）vs 認可（Authorization）
- OAuth 2.0 / OpenID Connect の概要
- OWASP Top 10の概要
- セキュリティを意識したアーキテクチャ（ゼロトラスト、最小権限の原則）

**追加方法:** `docs/security-basics.md` として補足資料化

### 3-5. API設計（検討）

**なぜ必要か:** マイクロサービス回でAPI通信に触れるが、API自体の設計原則（命名規則、バージョニング、エラーハンドリング）を体系的に学ぶ機会がない。

**カバーすべき内容:**
- REST API設計の原則（リソース指向、ステータスコード、HATEOAS）
- GraphQL — いつ・なぜ選ぶか
- gRPC — マイクロサービス間通信での利点
- API バージョニング戦略
- OpenAPI（Swagger）によるAPI定義

**追加方法:** 第7回（マイクロサービス）の補足資料として追加

## 4. 優先度マトリクス

### すぐ取り組むべき（効果: 高 / 工数: 中）
1. **一貫したサンプルプロジェクトの設計** — カリキュラム全体の「線」のつながりを大幅に強化
2. **実行可能な演習環境の整備** — 「見るだけ」から「手を動かす」への転換
3. **課題の模範解答** — 自学自習の質を向上

### 次のイテレーションで（効果: 中〜高 / 工数: 中）
4. **テスト戦略セクション** — クリーンアーキテクチャの価値を実感させる
5. **実務適用シナリオ** — 「明日から使える」感を強化
6. **中間振り返り** — 脱落防止と理解度の確認
7. **アジャイル開発手法との関連** — 「完璧な設計を最初から」という誤解を防ぐ

### 余裕ができたら（効果: 中 / 工数: 低〜中）
8. 可観測性セクション
9. セキュリティ基礎資料
10. API設計ガイド
11. 学習成果アセスメントテンプレート
