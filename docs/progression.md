## 各テーマのつながり詳細

### オブジェクト指向 → デザインパターン / DDD / クリーンアーキテクチャ

オブジェクト指向のSOLID原則（特に**依存性逆転の原則**と**インターフェース分離の原則**）は、後続のすべてのテーマの土台になります。

- **デザインパターン**: GoFパターンはオブジェクト指向の設計テクニックそのもの
- **DDD**: エンティティ、値オブジェクト、集約はオブジェクト指向のクラス設計が前提
- **クリーンアーキテクチャ**: 依存性逆転の原則がレイヤー間の依存制御の核心

### RDB / データモデリング → DDD / マイクロサービス

データベース設計の知識は、ドメインモデルの永続化やサービス分割の判断に直結します。

- **DDD**: リポジトリパターンによるドメインモデルの永続化設計
- **マイクロサービス**: データベース分割（Database per Service）の設計判断

### デザインパターン → DDD / クリーンアーキテクチャ

デザインパターンの知識があることで、DDDやクリーンアーキテクチャの実装が具体的にイメージできます。

- **DDD**: Repository、Factory、Strategy パターンが直接活用される
- **クリーンアーキテクチャ**: Adapter、Gateway パターンで外部との境界を設計

### DDD → クリーンアーキテクチャ → マイクロサービス

この3つは密接に関連し、スケールが段階的に大きくなります。

- **DDD**: 1つのドメイン内の構造化（戦術的設計）
- **クリーンアーキテクチャ**: 1つのアプリケーション内のレイヤー構造
- **マイクロサービス**: 複数サービス間の分割と連携（DDDの戦略的設計が活きる）

### マイクロサービス → オーケストレーション / イベント駆動

マイクロサービスを実際に運用する際の2つの主要なアプローチです。

- **オーケストレーション**: サービスのデプロイ・スケーリング・管理の仕組み
- **イベント駆動**: サービス間の非同期通信とデータ整合性の確保

## 各回で持ち帰ってほしい核心メッセージ

| 回 | テーマ | 核心メッセージ |
|----|--------|---------------|
| 第1回 | オブジェクト指向 | 「変更に強いコードは、責務が明確で依存が制御されている」 |
| 第2回 | RDB | 「データの整合性を守る仕組みを理解し、適切に活用する」 |
| 第3回 | データモデリング | 「良いデータモデルは、ビジネスの構造を正しく反映している」 |
| 第4回 | デザインパターン | 「パターンは道具。問題に合わせて選択し、過剰適用を避ける」 |
| 第5回 | DDD | 「ビジネスの言葉でコードを書くことが、最も強力な設計手法」 |
| 第6回 | クリーンアーキテクチャ | 「ビジネスロジックを外部の詳細から守ることが、変更容易性の鍵」 |
| 第7回 | マイクロサービス | 「分割はコストを伴う。モノリスから始めて、必要に応じて分割する」 |
| 第8回 | オーケストレーション | 「インフラの自動化と標準化が、サービスのスケーラビリティを支える」 |
| 第9回 | イベント駆動 | 「非同期は強力だが複雑。結果整合性のトレードオフを理解する」 |
