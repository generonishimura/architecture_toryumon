# 講師・ファシリテーター準備ガイド

> 各回の講師・ファシリテーターが開催前に確認すべき準備事項をまとめたガイドです。

## 全体の心構え

[運営方針](overview.md#運営方針)に基づき、以下を意識してください。

- **「なぜ」を伝える** — 技術の使い方だけでなく、その設計が必要な理由を説明する
- **トレードオフを示す** — 「正解」を押し付けず、選択肢とその長所・短所を提示する
- **実務の具体例を交える** — 教科書的な説明に留まらず、現場での経験談を盛り込む
- **議論を促す** — 一方通行の講義にせず、参加者の発言を引き出す

## 講師の学びサイクル — 「教えることは最高の学び」

講師・ファシリテーターとしての準備は、単なる「講義の段取り」ではありません。**教えるために学ぶプロセス**そのものが、講師自身のスキルを飛躍的に高めます。

### 準備 → 講義 → 振り返りのサイクル

1. **準備フェーズ（学びの深化）** — テーマを「教えられるレベル」まで理解する。自分の言葉で説明できるか、コード例を即興で書けるかが基準。[自己学習ガイド](instructor-learning-guide.md)と[セルフチェックシート](instructor-self-check.md)を活用する
2. **講義フェーズ（学びの検証）** — 参加者の質問や反応が、自分の理解の浅い部分を明らかにしてくれる。「うまく説明できなかった箇所」こそが次の学びのヒント
3. **振り返りフェーズ（学びの定着）** — 開催後に改善メモを残し、次回の講師へ引き継ぐ。自分の理解が深まった点・まだ曖昧な点を記録する

### 講師同士の学び合い

- **他の回の聴講を推奨** — 自分の担当回以外にも参加することで、テーマ間のつながりが実感できる。特に前後の回は聴講する価値が高い
- **講師間ナレッジシェア** — 講義後に「参加者からこういう質問が出た」「この説明が刺さった」を共有する場（Slackチャンネル等）を設ける
- **ペア講師制度の検討** — 知識レベルAの回（第2, 5, 7, 8, 9回）では、経験者とファシリテーターのペアで担当することで、双方の学びになる

## 準備タイムライン

### 3週間前（知識レベルAの回のみ）

- [ ] 担当回のワークショップREADME（`workshops/XX-xxx/README.md`）を精読する
- [ ] [セルフチェックシート](instructor-self-check.md)の該当回を実施し、未チェック項目を洗い出す
- [ ] 実務経験の棚卸し — 成功体験・失敗体験をそれぞれ3つ以上リストアップする
- [ ] 不足している知識領域を特定し、[参考書籍クイックリファレンス](instructor-reading-list.md)から必読書を選定する
- [ ] ペア講師が必要かを判断し、必要であればアサインを調整する

### 2週間前

- [ ] 担当回のワークショップREADME（`workshops/XX-xxx/README.md`）を精読する（知識レベルB/Cの回はここから開始）
- [ ] 前回・次回の内容を確認し、つながりを意識する（[学習の流れ](progression.md)参照）
- [ ] [「よくある誤解」ハンドブック](common-misconceptions.md)の該当回を確認し、自分自身が誤解を持っていないか点検する
- [ ] 講義のアウトラインを作成する
- [ ] 参加者の事前アンケート（経験年数、業務で使用している技術、特に学びたいこと）を設計・配信する
- [ ] 前回講師からの引き継ぎメモ（参加者の理解度、盛り上がったトピック、持ち越し質問）を確認する
- [ ] 実務での具体的なエピソード（成功・失敗）を最低3つ準備し、講義の中で使えるように整理する

### 1週間前

- [ ] スライド・講義資料を作成する
- [ ] TypeScriptコード例を作成し、動作確認する
- [ ] ライブコーディング用のステップバイステップ台本を作成する（途中でエラーが出た場合の対応も含む）
- [ ] ディスカッション用の設計課題（1〜2問）を用意する
- [ ] 復習課題・コード課題を作成する
- [ ] 前回の課題の模範解答を準備する
- [ ] 参加者の事前アンケート結果を分析し、講義の力点を調整する
- [ ] [セルフチェックシート](instructor-self-check.md)を再度実施し、全項目にチェックが付くか確認する
- [ ] 想定質問リストを作成し、各質問への回答を準備する（[講師補助資料](../workshops/)の「想定される質問」を参照）
- [ ] 時間が余った場合の「発展トピック」を2〜3個用意する

### 3日前

- [ ] スライド・コード例の通しリハーサルを実施する（可能であれば他の講師に聞いてもらう）
- [ ] 時間配分を確認し、各パートの時間を実測する
- [ ] コード例をリポジトリにプッシュし、参加者がcloneして動かせる状態にする
- [ ] リモート参加者がいる場合、オンラインツールの設定（ブレイクアウトルーム、画面共有権限等）を確認する

### 前日

- [ ] スライド・コード例の最終確認
- [ ] 会議室 or オンライン会議ツールの動作確認
- [ ] デモ環境（Node.js、エディタ、ターミナル）の準備
- [ ] タイムテーブルの確認（下記参照）
- [ ] バックアップ計画の確認（デモが動かない場合の代替スライド、ネットワーク障害時の対応等）
- [ ] 体調とメンタルの確認 — 十分な睡眠を取り、翌日の集中力を確保する

### 当日（開始30分前）

- [ ] 画面共有・マイクの確認
- [ ] コード例をエディタで開いておく（フォントサイズを大きめに設定）
- [ ] ターミナルのフォントサイズ・配色を画面共有向けに調整する
- [ ] 前回の課題に対する参加者の提出状況を確認
- [ ] 飲み物・軽食を手元に準備する（2時間超の講義は体力を使う）
- [ ] 開始前の雑談トピック（アイスブレイク）を1つ用意しておく

## 各回の講師に求められる知識レベル

各回で扱う内容の深さが異なるため、講師に求められる準備量と前提知識も回ごとに変わります。以下の表を目安にしてください。

**知識レベルの定義:**
- **A（実務経験必須）** — そのテーマで実際にシステムを設計・運用した経験が求められる。書籍の知識だけでは参加者の「現場ではどうするのか」という質問に答えられない
- **B（深い理解が必要）** — 書籍や個人プロジェクトで十分に学習し、コード例を自力で書ける。実務経験があれば理想的だが、必須ではない
- **C（基本を押さえていればOK）** — テーマの基本概念を理解し、READMEの内容を参加者に説明できるレベル

### 一覧

| 回 | テーマ | 知識レベル | 講師の前提知識 | 準備の重点 |
|----|--------|:----------:|---------------|-----------|
| 第0回 | オリエンテーション | C | 全11回の全体像を把握していること | 参加者の緊張をほぐすファシリテーション力 |
| 第1回 | オブジェクト指向 | B | SOLID原則を自分の言葉で説明でき、違反例・改善例をコードで示せる | TypeScriptでのBad/Goodコード例の準備 |
| 第2回 | RDB | A | 正規化・インデックス・トランザクション分離レベルを実務で扱った経験 | スロークエリやデッドロックなど現場で起きる問題の実例 |
| 第3回 | データモデリング | B | ER図の作成経験、概念/論理/物理モデルの使い分けを理解している | 参加者が取り組みやすい題材でのモデリング演習準備 |
| 第4回 | デザインパターン | B | GoFパターンのうち実務で頻出するもの（Strategy, Observer, Repository, Factory等）を実装できる | 「どんな場面で使うか」「使いすぎの弊害」の具体例 |
| 第5回 | DDD | A | エンティティ・値オブジェクト・集約の設計経験。境界づけられたコンテキストの分割を実務で検討した経験 | ユビキタス言語の策定や集約設計で苦労した実体験 |
| 第6回 | クリーンアーキテクチャ | B | レイヤードアーキテクチャやヘキサゴナルアーキテクチャの構成を理解し、TypeScriptで依存性逆転を含むプロジェクト構成を書ける | 「やりすぎ」と「ちょうどいい」のラインを示す実例 |
| 第7回 | マイクロサービス | A | モノリスからの分割、またはマイクロサービスの設計・運用経験。サービス間通信やデータ分割で発生する問題を肌感覚で知っている | モノリスとの比較、分割判断の失敗談・成功談 |
| 第8回 | オーケストレーション | A | Docker/Kubernetes の実運用経験。CI/CDパイプラインの構築経験 | デモ環境（ローカルK8s等）の構築、参加者が手を動かせる準備 |
| 第9回 | イベント駆動 | A | メッセージブローカー（Kafka, RabbitMQ等）を使ったシステムの設計・運用経験。CQRS/Sagaの実装経験があれば理想的 | 結果整合性やメッセージ順序保証の具体的なトラブル事例 |
| 第10回 | 総合演習 | B | 第1〜9回の全テーマの概要を把握。グループ演習のファシリテーション能力 | 演習題材の要件設計、グループへの適切な介入タイミング |

### 知識レベル別の推奨準備期間

| レベル | 推奨準備期間 | 具体的な準備 |
|:------:|:----------:|-------------|
| A | 2〜3週間前から | README精読 → 実務経験の棚卸し → 具体的なエピソード3つ以上をメモ → コード例作成 |
| B | 2週間前から | README精読 → 参考書籍1冊を通読 or 再読 → コード例を自力で書いて動作確認 |
| C | 1週間前から | README精読 → 全体の流れを把握 → 進行台本の作成 |

### 講師のアサイン指針

- **Aレベルの回（第2, 5, 7, 8, 9回）** は、そのテーマの実務経験者をアサインする。経験者がいない場合は、社外ゲスト講師やペア講師（知識担当 + ファシリテーション担当）の体制を検討する
- **Bレベルの回（第1, 3, 4, 6, 10回）** は、学習意欲の高いメンバーが準備期間をしっかり確保すれば担当可能。ただし参考書籍の通読は必須
- **Cレベルの回（第0回）** は、企画全体を把握しているオーガナイザーが適任

## フェーズ別の講義のコツ

### フェーズ1: 基礎編（第0〜3回）

参加者の前提知識にばらつきがある時期です。

- 用語の定義を丁寧に行う（後のフェーズで共通言語として使うため）
- 「知っている人」と「初めての人」の差が出やすいので、知っている人にはファシリテーター役を任せる
- コード例は短く、1つの概念に集中する

### フェーズ2: 設計パターン編（第4〜6回）

フェーズ1の知識を前提に、より抽象的な概念を扱います。

- フェーズ1で学んだ内容との関連を明示する（例：「第1回のSOLID原則のうち、依存性逆転がここで再登場します」）
- パターンの「適用すべき場面」と「適用すべきでない場面」の両方を示す
- 実務のコードに近い、やや複雑なコード例を用意する

### フェーズ3: 分散システム編（第7〜10回）

概念が大きくなり、コード例だけでは伝えにくくなります。

- アーキテクチャ図・シーケンス図を積極的に使う
- 「モノリスとの比較」を常に意識し、トレードオフを明確にする
- 可能であれば、実際のシステム構成例（社内 or 公開事例）を紹介する

## ディスカッションの進め方

ディスカッションは参加者の理解を深める重要な時間です。

### 設計課題の作り方

- **正解が1つではない問い**を設定する（例：「この要件に対して、どのパターンを適用するか？」）
- 要件を簡潔に記述し、制約条件を2〜3個付ける
- 参加者が5分以内に議論を始められる程度の具体性を持たせる

### ファシリテーションのポイント

- まず個人で1〜2分考える時間を取る
- 小グループ（2〜3人）で議論させてから全体共有する
- 「なぜその選択をしたか」を言語化させる
- 異なる意見が出たら、それぞれのトレードオフを整理する

## 課題の作成ガイド

[課題の運用](schedule.md#課題の運用)に沿って、以下の3種類を用意します。

| 種類 | 目的 | 目安 |
|------|------|------|
| 復習問題（3〜5問） | 講義内容の定着確認 | 15分で回答可能 |
| コード課題 | テーマの実践的な理解 | 30〜60分で取り組める規模 |
| 発展課題（任意） | より深い探究 | 60分〜、意欲のある参加者向け |

### 課題作成のポイント

- 復習問題は講義スライドを見返せば答えられるレベルにする
- コード課題はテンプレートやスターターコードを用意し、参加者がゼロから書かなくてよいようにする
- 発展課題は「答えがない問い」でもよい（設計判断を言語化させる）

## 参加者のレディネス（準備状態）把握

講義の効果を最大化するためには、参加者の現在地を正確に把握することが不可欠です。

### 事前アンケートの設計

開催2週間前に以下の項目を含むアンケートを配信し、1週間前までに回収します。

| 項目 | 質問例 | 目的 |
|------|--------|------|
| 経験年数 | 「エンジニアとしての実務経験は何年ですか？」 | 全体の経験レベルの把握 |
| 技術スタック | 「現在の業務で使用している言語・フレームワークは？」 | コード例の文脈調整 |
| テーマとの関わり | 「今回のテーマ（例：DDD）を実務で使ったことがありますか？」 | 説明の深さ調整 |
| 学びの期待 | 「この回で最も学びたいことは何ですか？」 | ディスカッション課題の調整 |
| 前回の理解度 | 「前回の内容で理解が不十分だと感じる箇所はありますか？」 | 振り返りパートの調整 |

### アンケート結果に基づく講義調整

| 参加者の傾向 | 講義の調整方法 |
|---|---|
| テーマの実務経験者が多い | 基本説明を短縮し、応用・ディスカッションに時間を割く。経験者に事例共有を依頼する |
| テーマの初学者が多い | 基本概念の説明を丁寧に行い、コード例を増やす。専門用語は必ず定義してから使う |
| 経験レベルにばらつきがある | 基本パートは全員向け、応用パートでは経験者をサブファシリテーターとして活用する |
| 前回の理解が不十分な参加者がいる | 振り返りパートを10分に延長し、前回のキーポイントを補足する |

### 参加者タイプ別の対応戦略

講義中にはさまざまなタイプの参加者がいます。事前に対応方針を考えておくことで、当日のファシリテーションがスムーズになります。

| タイプ | 特徴 | 対応方法 |
|--------|------|----------|
| 積極的な質問者 | 次々と質問し、他の参加者の発言機会を奪いがち | 「良い質問ですね。他の方はどう思いますか？」と議論を広げる |
| 沈黙型 | 理解しているが発言しない | 小グループでの議論を通じて発言の機会を作る。直接指名は避ける |
| 懐疑的な参加者 | 「それは現場では使えない」と否定的 | 「どういう場面で使えないと感じますか？」と具体化させ、トレードオフの議論に転換する |
| 先走り型 | 後の回で扱う内容に触れたがる | 「まさにそのポイントは第X回で詳しく扱います」と予告して、今回のスコープに戻す |

## リモート / ハイブリッド開催の準備

対面とリモート（またはその混在）では、ファシリテーションの方法を変える必要があります。

### リモート開催時の追加準備

#### 環境・ツール

- [ ] ビデオ会議ツール（Zoom/Teams/Meet等）のブレイクアウトルーム機能を確認・テストする
- [ ] 画面共有時のフォントサイズを通常の1.5〜2倍に設定する（参加者のモニターサイズは様々）
- [ ] コード例を事前にGitHubリポジトリまたはGistで共有し、参加者が手元で見られるようにする
- [ ] バーチャルホワイトボード（Miro、FigJam等）を準備する（ER図やアーキテクチャ図の共同作業用）
- [ ] チャット欄での質問受付ルール（いつでもOK / 区切りで）を決めておく
- [ ] 録画の可否・共有範囲を事前に参加者と合意する

#### ファシリテーション

- [ ] 参加者にカメラONを推奨する（反応が見えないと講師のペース調整が困難）
- [ ] 5分ごとに「ここまでで質問はありますか？」と確認するリズムを意識する
- [ ] リアクション機能（👍、✋等）の使い方を最初に説明する
- [ ] ディスカッションはブレイクアウトルーム（2〜3人）で行い、全体共有は代表者1名に発表してもらう
- [ ] 休憩は対面より長めに取る（画面疲れ対策として10分→15分）

### ハイブリッド開催時の追加注意点

- [ ] リモート参加者の音声が会議室で聞こえるか、逆に会議室の音声がリモートで聞こえるかを事前テストする
- [ ] 会議室のホワイトボードを使う場合は、カメラで映す or バーチャルホワイトボードと併用する
- [ ] ディスカッションのグループ分けで、リモート参加者だけのグループを避ける（対面とリモートを混在させる）
- [ ] リモート参加者が「置いていかれた感」を持たないよう、チャットでのフォローアップを意識する

## 多角的な準備の観点

### 観点1: ビジネスインパクトの視点

技術的な正しさだけでなく、「なぜこの設計がビジネスに影響するか」を語れるよう準備します。

- **各テーマのビジネス価値を整理する** — 例：クリーンアーキテクチャ → 技術負債の抑制 → 機能追加のリードタイム短縮 → ビジネスの俊敏性向上
- **コスト観点でのトレードオフ** — 設計にかける時間 vs 将来の保守コスト、チーム規模による採用基準の違い
- **実際のプロジェクト事例** — 設計判断がプロジェクトの成否にどう影響したか（可能であれば自社事例、なければ公開事例）

### 観点2: チーム・組織の視点

アーキテクチャは個人の技術だけでなく、チームや組織のあり方に大きく影響します。

- **チームトポロジーとの関連** — マイクロサービスの分割とチーム構成は密接に関連する（コンウェイの法則）
- **コミュニケーションコスト** — アーキテクチャの境界がチーム間のコミュニケーション境界と一致しないときの問題
- **段階的な導入戦略** — 「いきなり理想の設計」ではなく、チームの成熟度に合わせた段階的な改善の考え方
- **コードレビューの観点** — 学んだ設計原則をコードレビューでどう活用するか

### 観点3: キャリア・成長の視点

参加者にとって、アーキテクチャの学習が自身のキャリアにどうつながるかも重要な視点です。

- **設計力が求められる場面** — テックリード、アーキテクト、シニアエンジニアへのステップアップに不可欠
- **「なぜそうするか」を説明する力** — 設計判断の根拠を言語化できることが、チームの信頼獲得につながる
- **T字型スキルの深化** — 幅広い技術知識の上に、設計という縦の軸を持つことの価値

### 観点4: 技術トレンドの視点

各テーマの現在の位置づけとトレンドを把握しておきます。

| 回 | 押さえるべきトレンド |
|----|---------------------|
| 第1回 | 関数型プログラミングとの融合、TypeScriptの型システムを活かしたOOP |
| 第2回 | NewSQL（CockroachDB, TiDB等）の台頭、サーバーレスDB |
| 第3回 | GraphQLスキーマとデータモデルの関係、ORMの進化（Prisma, Drizzle） |
| 第4回 | 関数型パターン（Monad, Functor等）の実務適用、React Hooksとパターンの関係 |
| 第5回 | イベントストーミング、EventModeling等の新しいモデリング手法 |
| 第6回 | モジュラーモノリス、Feature-Slicedアーキテクチャ |
| 第7回 | サーバーレスアーキテクチャ、エッジコンピューティング |
| 第8回 | GitOps、Platform Engineering、IaC（Terraform/Pulumi）の標準化 |
| 第9回 | CloudEvents仕様、Kafka vs NATS vs Pulsar の比較 |
| 第10回 | AI/LLMを活用した設計支援、アーキテクチャフィットネス関数 |

### 観点5: 失敗学の視点

各テーマで起きやすい失敗パターンを整理し、講義に盛り込みます。「失敗から学ぶ」は参加者の記憶に最も残りやすい教材です。

| 回 | よくある失敗パターン | 講義での活かし方 |
|----|---------------------|-----------------|
| 第1回 | SOLID原則の教条的な適用で、簡単な処理まで過剰に抽象化 | 「原則は道具であって目的ではない」を冒頭で伝える |
| 第2回 | インデックスの貼りすぎで書き込み性能が劣化 | EXPLAIN結果を見せながら「インデックスのコスト」を実感させる |
| 第3回 | 概念モデルを飛ばしていきなりテーブル設計し、業務の構造を見失う | 3段階モデリングのプロセスを体験させる |
| 第4回 | 「パターン病」— 全てのコードにパターンを適用しようとする | 「パターンが不要な場面」を具体的に示す |
| 第5回 | 集約を大きくしすぎてパフォーマンスが劣化、集約間の整合性が取れない | 集約のサイズ判断の基準を繰り返し強調する |
| 第6回 | レイヤーを機械的に分けただけで、依存の方向が制御されていない | ディレクトリ構造だけでなく、import文の方向を確認するデモ |
| 第7回 | 分散モノリス — サービスを分けたのに結合度が高いまま | 「分割の判断基準」を具体的に、DDDのBounded Contextと紐付けて説明 |
| 第8回 | 「Kubernetesを使えばスケールする」という誤解でオーバーエンジニアリング | Docker Composeで十分な場面とK8sが必要な場面を明確に線引き |
| 第9回 | 冪等性を考慮せず、メッセージ重複で不整合が発生 | 冪等性の確保パターンをコードで具体的に示す |
| 第10回 | 技術選定を「流行っているから」で判断し、チームの能力や要件に合わない | トレードオフマトリクスを使った判断フレームワークを体験させる |

## トラブルシューティング

### 講義進行に関するトラブル

| 状況 | 対処法 |
|------|--------|
| 時間が足りない | ディスカッションを短縮し、Slackなどで非同期継続する。次回冒頭で補足する |
| 時間が余る | 準備していた発展トピックを紹介する / 参加者の実務課題を議論する |
| 質問に答えられない | 正直に「次回までに調べます」と伝え、必ずフォローアップする |
| 参加者の理解度に差がある | 理解が早い人にサブファシリテーター役を依頼する |
| 議論が盛り上がらない | 講師自身が「自分ならこう考える」と口火を切る / 具体的なシナリオを提示する |
| 特定の参加者が話しすぎる | 「ありがとうございます。他の方の意見も聞いてみましょう」と自然に転換する |
| 質問が脱線する | 「面白い観点ですね。これは第X回で扱う内容に近いので、そちらで詳しく議論しましょう」 |
| 参加者の集中力が切れてきた | 休憩を前倒しにする / 短いハンズオン（2〜3分のコーディング）を挟む |

### 技術的なトラブル

| 状況 | 対処法 |
|------|--------|
| ライブコーディングでエラーが出る | 「よくあるエラーですね。一緒にデバッグしましょう」と学びの機会に変える。事前にバックアップコードを用意しておく |
| デモ環境が動かない | スクリーンショットやスライドに切り替える。コード例はGitHubで共有し、参加者が後で手元で動かせるようにする |
| ネットワーク障害（リモート開催時） | 事前にスライドPDFを参加者に共有しておく。講師が切断された場合に備え、サブ講師 or 運営が進行を引き継ぐ手順を決めておく |
| 画面共有ができない | 代替手段（チャットでコード共有、参加者のローカルでリポジトリを開いてもらう）に切り替える |

## 講師のメンタル準備

技術的な準備だけでなく、講義に臨む心構えも大切です。

### 完璧を目指さない

- 「全ての質問に完璧に答える」ことは目標ではない。「参加者が自分で考え始めるきっかけを作る」ことが目標
- 分からないことは正直に認め、一緒に調べる姿勢が参加者の信頼を得る
- 初めて講師を担当する場合、緊張するのは自然なこと。最初の5分を乗り切ればペースがつかめる

### エネルギー管理

- 2時間超の講義は想像以上に体力・集中力を使う。前日は十分な睡眠を取る
- 講義前に5分間の深呼吸や軽いストレッチを行い、リラックスする
- 休憩時間は自分自身の休憩にも充てる（参加者の質問対応は手短にする）

### フィードバックとの向き合い方

- ネガティブなフィードバックは改善の材料として受け止め、個人攻撃と捉えない
- 次回の講師に改善点を伝えることで、勉強会全体の質が向上する
- 良かった点も記録し、自信を持って次の機会に活かす
