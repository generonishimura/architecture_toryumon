# 講師・ファシリテーター準備ガイド

> 各回の講師・ファシリテーターが開催前に確認すべき準備事項をまとめたガイドです。

## 全体の心構え

[運営方針](overview.md#運営方針)に基づき、以下を意識してください。

- **「なぜ」を伝える** — 技術の使い方だけでなく、その設計が必要な理由を説明する
- **トレードオフを示す** — 「正解」を押し付けず、選択肢とその長所・短所を提示する
- **実務の具体例を交える** — 教科書的な説明に留まらず、現場での経験談を盛り込む
- **議論を促す** — 一方通行の講義にせず、参加者の発言を引き出す

## 各回の講師に求められる知識レベル

各回で扱う内容の深さが異なるため、講師に求められる準備量と前提知識も回ごとに変わります。以下の表を目安にしてください。

**知識レベルの定義:**
- **A（実務経験必須）** — そのテーマで実際にシステムを設計・運用した経験が求められる。書籍の知識だけでは参加者の「現場ではどうするのか」という質問に答えられない
- **B（深い理解が必要）** — 書籍や個人プロジェクトで十分に学習し、コード例を自力で書ける。実務経験があれば理想的だが、必須ではない
- **C（基本を押さえていればOK）** — テーマの基本概念を理解し、READMEの内容を参加者に説明できるレベル

### 知識レベル別の推奨準備期間

| レベル | 推奨準備期間 | 具体的な準備 |
|:------:|:----------:|-------------|
| A | 2〜3週間前から | README精読 → 実務経験の棚卸し → 具体的なエピソード3つ以上をメモ → コード例作成 |
| B | 2週間前から | README精読 → 参考書籍1冊を通読 or 再読 → コード例を自力で書いて動作確認 |
| C | 1週間前から | README精読 → 全体の流れを把握 → 進行台本の作成 |

### 講師のアサイン指針

- **Aレベルの回（第2, 5, 7, 8, 9回）** は、そのテーマの実務経験者をアサインする。経験者がいない場合は、社外ゲスト講師やペア講師（知識担当 + ファシリテーション担当）の体制を検討する
- **Bレベルの回（第1, 3, 4, 6, 10回）** は、学習意欲の高いメンバーが準備期間をしっかり確保すれば担当可能。ただし参考書籍の通読は必須
- **Cレベルの回（第0回）** は、企画全体を把握しているオーガナイザーが適任

## フェーズ別の講義のコツ

### フェーズ1: 基礎編（第0〜3回）

参加者の前提知識にばらつきがある時期です。

- 用語の定義を丁寧に行う（後のフェーズで共通言語として使うため）
- 「知っている人」と「初めての人」の差が出やすいので、知っている人にはファシリテーター役を任せる
- コード例は短く、1つの概念に集中する

### フェーズ2: 設計パターン編（第4〜6回）

フェーズ1の知識を前提に、より抽象的な概念を扱います。

- フェーズ1で学んだ内容との関連を明示する（例：「第1回のSOLID原則のうち、依存性逆転がここで再登場します」）
- パターンの「適用すべき場面」と「適用すべきでない場面」の両方を示す
- 実務のコードに近い、やや複雑なコード例を用意する

### フェーズ3: 分散システム編（第7〜10回）

概念が大きくなり、コード例だけでは伝えにくくなります。

- アーキテクチャ図・シーケンス図を積極的に使う
- 「モノリスとの比較」を常に意識し、トレードオフを明確にする
- 可能であれば、実際のシステム構成例（社内 or 公開事例）を紹介する

## ディスカッションの進め方

ディスカッションは参加者の理解を深める重要な時間です。

### 設計課題の作り方

- **正解が1つではない問い**を設定する（例：「この要件に対して、どのパターンを適用するか？」）
- 要件を簡潔に記述し、制約条件を2〜3個付ける
- 参加者が5分以内に議論を始められる程度の具体性を持たせる

### ファシリテーションのポイント

- まず個人で1〜2分考える時間を取る
- 小グループ（2〜3人）で議論させてから全体共有する
- 「なぜその選択をしたか」を言語化させる
- 異なる意見が出たら、それぞれのトレードオフを整理する

## 課題の作成ガイド

[課題の運用](schedule.md#課題の運用)に沿って、以下の3種類を用意します。

| 種類 | 目的 | 目安 |
|------|------|------|
| 復習問題（3〜5問） | 講義内容の定着確認 | 15分で回答可能 |
| コード課題 | テーマの実践的な理解 | 30〜60分で取り組める規模 |
| 発展課題（任意） | より深い探究 | 60分〜、意欲のある参加者向け |

### 課題作成のポイント

- 復習問題は講義スライドを見返せば答えられるレベルにする
- コード課題はテンプレートやスターターコードを用意し、参加者がゼロから書かなくてよいようにする
- 発展課題は「答えがない問い」でもよい（設計判断を言語化させる）

## 開催後のアクション

- [ ] 参加者フィードバック（振り返りフォーム）を収集する
- [ ] スライド・コード例をリポジトリにアップロードする
- [ ] 次回の講師へ引き継ぎ事項を共有する（参加者の理解度、盛り上がったトピック等）
- [ ] 改善メモを残す（時間配分の過不足、説明が難しかった箇所等）

## トラブルシューティング

| 状況 | 対処法 |
|------|--------|
| 時間が足りない | ディスカッションを短縮し、Slackなどで非同期継続する |
| 時間が余る | 準備していた発展トピックを紹介する / 参加者の実務課題を議論する |
| 質問に答えられない | 正直に「次回までに調べます」と伝え、必ずフォローアップする |
| 参加者の理解度に差がある | 理解が早い人にサブファシリテーター役を依頼する |
| 議論が盛り上がらない | 講師自身が「自分ならこう考える」と口火を切る / 具体的なシナリオを提示する |
