# 講師の理解度セルフチェックシート

> 「教えられるレベル」は「知っているレベル」よりも遥かに深い理解を必要とする。本シートは、各回の講師を担当する前に自分自身の理解度を客観的に確認するためのものである。チェックが付かない項目があれば、該当トピックを重点的に復習してから講義に臨むこと。

## 使い方

1. 担当する回のセクションを開き、各項目に対して正直にチェックする。
2. チェックできない項目は、講義前に必ず復習・準備を行う。
3. 全項目にチェックが付いた状態を「講義可能な最低ライン」とする。
4. 可能であれば、他の講師やメンバーに対して口頭で説明し、フィードバックを受ける。

## 第0回：オリエンテーション

### 概念の理解
- [ ] ソフトウェアアーキテクチャとは何か、なぜ重要なのかを、具体的な失敗例を交えて説明できるか
- [ ] 全11回の講座全体像について、各回の関連性と学習の流れを論理的に説明できるか

### 実践力
- [ ] 受講者のスキルレベルに応じて、講座のゴールイメージを具体的なアウトプット例とともに提示できるか
- [ ] 実際のプロジェクトで「アーキテクチャの判断ミスがどのような結果を招いたか」を自身の経験から語れるか

### 質問対応力
- [ ] 「アーキテクチャを学ぶより、フレームワークの使い方を学ぶ方が実務に直結するのでは」と聞かれたら、両者の関係を説明できるか
- [ ] 「経験が浅い段階でアーキテクチャを学ぶ意味はあるのか」と聞かれたら、段階的な学習の価値を説明できるか

## 第1回：オブジェクト指向

### 概念の理解
- [ ] SOLID原則の各原則（SRP, OCP, LSP, ISP, DIP）を、比喩を使わずに技術的に正確に説明できるか

## 第2回：RDB

### 概念の理解
- [ ] 正規化（第1〜第3正規形、BCNF）の各段階の定義と、なぜその正規形が必要かを説明できるか
- [ ] ACID特性の各要素が具体的にどのような問題を防ぐのかを、トランザクション障害のシナリオで説明できるか
- [ ] インデックスの内部構造（B-Tree）と、なぜ検索が高速化されるのかを原理から説明できるか

### 実践力
- [ ] 非正規化されたテーブルを段階的に第3正規形まで正規化する過程を、ホワイトボードで実演できるか
- [ ] スロークエリに対してEXPLAINの結果を読み解き、適切なインデックスを提案できるか

### 質問対応力
- [ ] 「正規化しすぎるとJOINが増えてパフォーマンスが落ちるのでは」と聞かれたら、正規化と非正規化の判断基準を説明できるか
- [ ] 「NoSQLではACIDを諦めているのか」と聞かれたら、CAP定理との関係を含めて説明できるか

## チェック結果の活用

### チェック状況に応じた対応

| チェック状況 | 対応 |
|---|---|
| 全項目チェック済み | 講義可能。想定外の質問に備えて関連トピックも確認する |
| 1〜2項目が未チェック | 該当項目を重点的に復習してから講義に臨む |
| 3項目以上が未チェック | 講義の延期を検討し、十分な準備期間を確保する |

### 復習の進め方

1. 未チェック項目について、まず自分の言葉で説明を書き出す（書けない部分が理解の穴である）
2. 書籍や公式ドキュメントで該当箇所を確認し、説明を修正する
3. 他者に口頭で説明し、質問を受けてみる（説明できない箇所を特定する）
4. 再度セルフチェックを行い、全項目にチェックが付くまで繰り返す

### 注意事項

- チェックが付く基準は「知っている」ではなく「受講者に対してその場で説明・実演できる」である
- 本シートは最低ラインの確認であり、チェックが全て付いたとしても追加の準備を怠らないこと
- 各回の講義後に、実際に受けた質問を追記して、次回以降の講師のためにシートを更新すること
